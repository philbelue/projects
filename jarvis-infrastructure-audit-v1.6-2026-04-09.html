Anomalies Resolved

Eight issues detected and corrected during this audit cycle. No outstanding critical or high-severity items.

01
Cron Scheduler Crash — runningAtMs TypeError

Nightly historian and all cron jobs were non-functional. OpenClaw 4.3 bug: reads job.state.runningAtMs before state initialization on disk-loaded jobs. Added "state": {} to each job in jobs.json.

02
Expired One-Time Cron Job

Stale deal-strategy-reminder-20260405 blocking scheduler. Removed — only nightly historian remains.

03
Auth Profiles Key Desync

auth-profiles.json held a stale Anthropic API key divergent from openclaw.json. Synced both files. Root cause of hot-reload auth failures.

04
LaunchD KeepAlive Policy

Plist used SuccessfulExit: false — only restarts on crashes. SIGTERM = clean exit = no restart. Changed to KeepAlive: true for unconditional recovery.

05
Docker Path References in Workspace

HEARTBEAT.md and DONT-LET-PHIL-CRASH-OUT.md still pointed to /home/node/.openclaw/. Updated to native paths.

06
Missing Cron State File

No state.json in cron directory. Created with proper initial structure for job execution tracking.

07
Gateway Port Conflict

Manual process holding port 18789 when launchd attempted second instance. Resolved — launchd is now sole process manager.

08
Docker Path Sweep

Global find/sed across all .openclaw/ JSON files. Zero /home/node references remaining.

Strategic Considerations

Best practices you haven't implemented yet, and how to leverage the full Claude + Jarvis dual-operator model. Think of it this way: Jarvis is your always-on operator. Claude is your on-demand architect. Together, they cover both real-time ops and deep strategic work.

Infrastructure Best Practices

!
Git-Versioned Config Backups

Initialize a git repo at ~/.openclaw/ and ~/.hermes/. Auto-commit nightly via cron. When a config change breaks something (it will), you get instant rollback with git diff showing exactly what changed. This is the single highest-leverage resilience move you're not doing yet.

!
Automated Nightly Backup to 4TB

You have 3.58 TB sitting idle. Set up a cron job that rsyncs ~/.openclaw/, ~/.hermes/, n8n database, and launchd plists to /Volumes/Jarvis/backups/$(date) nightly with 7-day rotation. If the internal SSD dies, you rebuild in under an hour instead of a day.

!
Disaster Recovery Runbook

One markdown file: /Volumes/Jarvis/REBUILD.md. Documents the exact sequence to rebuild the entire stack from scratch on a new machine using only the 4TB backup. Brew installs, plist placements, config restores, API key locations, service start order. You write it once while it's fresh. You'll thank yourself at 2 AM when something goes sideways.

!
Separate API Keys Per Service

OpenClaw and Hermes currently share the same Anthropic and OpenRouter keys. Create separate keys for each. Why: per-service usage auditing, independent revocation if one is compromised, and clean billing visibility. Takes 5 minutes, eliminates a class of debugging pain.

!
Log Rotation & Centralized Logging

Watchdog logs, OpenClaw stdout, Hermes output, n8n logs — these will grow unbounded. Set up newsyslog or a simple logrotate cron to rotate logs weekly, keep 4 weeks. Bonus: pipe all service logs to a single ~/.jarvis/logs/ directory so Jarvis can search them as context.

!
UPS / Power Protection

Mac Studio + 4TB external NVMe enclosure = two devices that die simultaneously on power loss. A $150 UPS gives you clean shutdown time and protects the NVMe from corruption mid-write. Especially matters now that Ollama models live on the external drive.

!
Pin Ollama Model Versions

Right now you pulled mistral-small3.1:latest. If Ollama auto-updates the tag, your local inference behavior changes without warning. Pin to a specific digest hash in your docs so you can always re-pull the exact version you tested against.

Your two AI layers serve fundamentally different roles. Jarvis is the 24/7 operator — always on, always listening, handling daily ops. Claude is the strategic partner you bring in for deep work, architecture, and high-stakes output. Here's how to maximize the combination:

Claude (Architect / Deep Work)

  • Infrastructure audits — periodic deep-dive sessions like this one. Jarvis handles daily health; Claude handles structural reviews.
  • n8n workflow design — architect complex multi-step workflows here, then deploy them through Jarvis. Claude designs the blueprint, Jarvis executes it.
  • Config debugging — when something breaks at the config level (like today's cron bug), bring the raw files here for source-code-level analysis.
  • Document generation — IXON proposals, one-pagers, deal strategy docs, interview prep materials. Artifact-quality output with file delivery.
  • Hermes skill authoring — write optimized skill definitions here, then inject them into Hermes. Claude writes the playbook, Hermes learns it.

  • Scheduled Cowork tasks — Claude can run on a schedule via Cowork. Morning briefing generation, weekly pipeline reviews, or periodic system audits — delivered as files to your desktop.

Jarvis (Operator / Always-On)

  • Real-time ops — Discord/Telegram interface, instant responses, daily monitoring, cron execution.
  • Nightly historian — automated daily logging, memory consolidation, context building.
  • Self-healing — cross-monitoring, service restarts, health checks every 120 seconds.

  • n8n trigger execution — runs the workflows Claude designs. Webhook triggers, API calls, data processing.
  • Proactive alerts — disk space warnings, service failures, API credit monitoring, anomaly detection.
  • Context accumulation — every conversation with Jarvis enriches MEMORY.md. Over time, Jarvis becomes the most context-rich agent in your stack.

The Handoff Pattern — Claude Designs, Jarvis Deploys

Use Claude to architect complex n8n workflows, cron jobs, Hermes skills, or config changes. Export them as JSON/YAML/code. Hand to Jarvis via Discord for deployment. You get Claude's depth AND Jarvis's persistence.

Jarvis accumulates daily context via historian + conversations. Periodically bring that context to Claude for strategic analysis: pipeline health, deal patterns, time allocation, system performance trends. Jarvis gathers the data, Claude finds the signal.
