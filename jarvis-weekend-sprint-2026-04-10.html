Decision Gates
Go / No-Go Checkpoints

If any gate fails, don’t push forward — fix the foundation. Building Hermes on an unstable base creates more problems than it solves.

Gate 1: Friday 6:30 PM — PM Recap

Pass: PM Recap fires and delivers coherent content to Discord. Proceed with hardening and scaffolding.

Fail: Debug the n8n workflow. Do hardening work (Keychain, backups, Eero) but hold off on Hermes scaffolding. Fix the workflow first.

Gate 2: Saturday 6:00 AM — AM Brief

Pass: AM Brief fires clean. Two automated deliveries in a row. Foundation is solid. Proceed with cold reboot + Hermes build.

Fail: Debug. Run the cold reboot test (valuable regardless). Build Hermes core but don’t deploy until AM Brief passes.

Gate 3: Sunday Afternoon — Hermes Foreground Test

Pass: Hermes runs in foreground, enters grace period, goes operational, health endpoint responds, Discord alert fires on simulated failure. Proceed to launchd deployment.

Fail: Fix the failing component. Don’t deploy to launchd until the foreground run is clean. A broken daemon in launchd is worse than no daemon — it restart-loops.

Gate 4: Monday 7:00 AM — Overnight Soak

Pass: No overnight Discord alerts. Healthchecks.io green. Hermes logs show clean cycles. System is operational.

Fail: Check Hermes error logs. Most likely: a health probe timeout threshold is too aggressive, or a service restarted and Hermes caught it (which is actually it doing its job). Tune config.yaml thresholds based on what you see.
