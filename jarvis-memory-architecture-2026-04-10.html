The Core Problem

Every Claude API call is stateless. When Jarvis gets a message in #ai-research, he has zero knowledge of what was discussed in #strategy 10 minutes ago. No conversation history. No shared context. No memory of past decisions. Each message is an island.

The Gap

What's missing between "Jarvis can talk" and "Jarvis remembers."

Current: Stateless Agent

  • Every API call starts from scratch
  • No recall of prior conversations
  • Can't connect discussions across channels
  • Obsidian vault is write-only (briefs write to it, nothing reads back)
  • Hermes model sitting idle on M4 Max
  • Context limited to current message + system prompt

Target: Context-Aware Agent

  • Retrieves relevant past conversations before responding
  • Knows what was decided in #strategy when answering in #ops
  • Obsidian vault is a live knowledge base — queried in real-time
  • Hermes handles summarization + embeddings locally (free)
  • Time-decay so recent context weighs more than old
  • Learns and improves over time
The 3 Missing Layers

1. Storage: A vector database to store embeddings of messages and knowledge. 2. Retrieval: A RAG pipeline that queries relevant context before each Claude call. 3. Processing: A local model (Hermes) that summarizes, embeds, and classifies — saving API costs and enabling background processing.

What Changes

Before Claude sees a message, n8n now runs a parallel retrieval: (1) embed the message via Hermes locally, (2) query the vector DB for similar past conversations, (3) query Obsidian vault for relevant knowledge, (4) grab the last ~20 messages from the conversation buffer. All of this gets merged into an enriched prompt. After Claude responds, the exchange gets embedded and stored back in the vector DB for future retrieval. This is the RAG loop.

Execution Plan

Recommended path: start with Option A to prove the concept, then layer in Option B. This is iterative — don't try to build everything at once.

Goal: Jarvis remembers the last 20 messages in each channel

This is the "get a half back dive down" rep. Minimal infrastructure, maximum proof of concept.

Add Memory Node to n8n Workflow

In the Jarvis Discord response workflow, add a ConversationBufferWindowMemory node before the Claude AI Agent node. Set window to 20 messages. n8n stores these in its built-in SQLite.

Easy · 30 min

Configure Session ID by Channel

Set the memory session ID to the Discord channel ID. This gives each channel its own conversation buffer. Jarvis now remembers recent messages per channel.

Easy · 15 min

Test Cross-Message Recall

Send Jarvis a message referencing something from 10 messages ago in the same channel. Verify he picks it up. That's your first win.

Easy · 15 min
Goal: Jarvis retrieves relevant context from ANY channel + Obsidian vault

This is the play action pass. Cross-channel awareness and knowledge base retrieval.

Deploy ChromaDB in Docker

Run ChromaDB on the Mac Studio. Single Docker container, ~500MB memory footprint. This becomes the semantic search layer for all stored conversations and vault content.

Medium · 1 hour

Set Up Hermes Embeddings via Ollama

Install nomic-embed-text (or similar) in Ollama alongside Hermes. This model converts text → vectors. Runs locally on M4 Max, $0 per embedding. n8n calls it via HTTP.

Medium · 1 hour

Build the Ingest Pipeline in n8n

New workflow: on every Discord message, (1) embed the message via Hermes/Ollama, (2) store the embedding + metadata (channel, user, timestamp) in ChromaDB. This runs in background — doesn't slow down responses.

Medium · 2 hours

Index Obsidian Vault

One-time workflow: read all markdown files from the Obsidian vault, chunk them (500-1000 tokens each), embed via Hermes, store in ChromaDB. Schedule re-indexing nightly to catch vault updates.

Medium · 2 hours

Add Retrieval Step to Response Workflow

Before calling Claude: embed the incoming message → query ChromaDB for top 5-10 similar past messages + vault chunks → inject as context in the prompt. n8n has a Vector Store Retriever node for this.

Medium · 2 hours

Test Cross-Channel Recall

Discuss a topic in #strategy, then ask about it in #ai-research without mentioning specifics. Jarvis should retrieve the #strategy context and connect the dots.

Easy · 15 min
Goal: Jarvis has autonomous long-term memory with self-summarization

The exotic formation reverse. Jarvis manages his own memory, compresses old context, and evolves.

Add Hermes Summarization Layer

When the conversation buffer exceeds 50 messages, trigger Hermes to summarize the oldest 30 into a compressed summary. Store the summary in ChromaDB and Obsidian. This is ConversationSummaryBufferMemory in practice.

Advanced · 3 hours

Implement Time-Decay Scoring

When retrieving from ChromaDB, weight results by recency: Score = Relevance × e^(-0.1 × days_old). Recent conversations rank higher than old ones. Prevents stale context pollution.

Advanced · 2 hours

Build "Server State" Summary

Nightly scheduled workflow: Hermes reads all channels' activity from the past 24 hours, generates a 500-word "server state" summary. This gets injected into every Claude call as persistent context. Jarvis always knows "what's going on."

Advanced · 3 hours

Vault Write-Back Loop

When Jarvis makes a decision or gives a recommendation, auto-write it back to the Obsidian vault under a "decisions" folder. This closes the loop — conversations become knowledge, knowledge informs future conversations.

Medium · 2 hours

Recommendation

Start Phase 1 today. It's a 1-hour rep that proves the concept. Phase 2 is your weekend project. Phase 3 is iterative and can happen over weeks.

Bottom Line

You have every piece of the stack already running on your machine. The gap isn't tools — it's the retrieval pipeline between them. ChromaDB + Hermes embeddings + n8n's vector store nodes close that gap. Total additional infrastructure cost: $0. This is all local compute on the M4 Max. The $100 in OpenRouter credit stays reserved for Claude reasoning — the expensive calls. Hermes handles the grunt work for free.
