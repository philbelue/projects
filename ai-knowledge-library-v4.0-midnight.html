🧬 Foundations & Core Concepts

Artificial Intelligence

AI
Foundations

The broad field of computer science focused on creating systems that can perform tasks typically requiring human intelligence. This includes learning from data, recognizing patterns, understanding language, and making decisions.

+ How it works
AI systems process input data through algorithms and models to produce outputs. The core idea: instead of hand-coding every rule, AI systems learn patterns from examples. This might be supervised learning (labeled data), unsupervised learning (finding hidden structure), or reinforcement learning (learning from rewards/penalties).
+ Why you care
Understanding AI as a spectrum—not magic—helps you make architectural decisions. Is this task best solved with ML? Should you build a pipeline or use a pre-trained model? How do you evaluate quality? Every AI implementation starts with this question.
⚙ In Jarvis
Jarvis is built on a stack of AI systems: Mistral Small 3.1 handles local chat on #local-chat, Gemini Flash synthesizes n8n workflows, Claude Sonnet tackles complex reasoning. Each has its AI sweet spot: Mistral for speed/privacy, Gemini for multi-modal synthesis, Claude for depth.

Machine Learning

ML
Foundations

A subset of AI where systems learn patterns from data rather than being explicitly programmed. ML algorithms adjust internal parameters (weights) to minimize errors on training examples, enabling generalization to new, unseen data.

+ How it works
You provide labeled or unlabeled data. The algorithm iteratively updates weights to reduce loss (prediction error). Common paradigms: supervised (predict Y from X using labeled data), unsupervised (find structure in unlabeled data), semi-supervised (mix of both), and reinforcement learning (learn from rewards).
+ Why you care
ML is the engineering discipline behind modern AI. Knowing the difference between supervised/unsupervised/RL shapes how you design systems. Do you have labeled data? Is real-time feedback available? These answers determine your approach.
⚙ In Jarvis
Jarvis doesn't train its own models—it sources them pre-trained and fine-tuned by others. But the ML principle is baked in: routing decisions (which model for which task?) are driven by learned patterns over 120 days of email/intent data. Pipeline Intelligence scans emails and learns to classify accounts as Active/Stalling/Cold/Closed.

Deep Learning

DL
Foundations

A subset of ML using neural networks with many layers (hence "deep"). These networks automatically learn hierarchical representations—low-level features in early layers, high-level abstractions in deeper layers—enabling powerful learning on complex data like images, text, and audio.

+ How it works
Each layer transforms its input through learned weights and biases, passing output to the next layer. Backpropagation computes gradients across all layers simultaneously. Deep networks can learn very complex patterns because shallow networks would require exponentially more parameters.
+ Why you care
All modern language models, image generators, and speech systems are deep learning. If you're working with large models or end-to-end learning on unstructured data, you're in deep learning territory. Understanding layer depth, parameters, and compute costs is critical.
⚙ In Jarvis
Mistral Small 3.1 (24B) is a deep neural network with ~24 billion parameters. Running it locally via Ollama in Q4_K_M quantization lets you leverage deep learning without cloud costs. For Gemini and Claude, deep learning is hidden—but you're still accessing deep networks managed by Anthropic and Google.

Neural Network

NN
Foundations

A computational graph composed of interconnected nodes (neurons) organized in layers. Each connection has a learnable weight; each neuron applies an activation function to its weighted sum of inputs. Neural networks are inspired by biological neurons but are mathematical abstractions.

+ How it works
Input layer → hidden layers → output layer. Each neuron computes: output = activation(Σ(weight × input) + bias). Activation functions (ReLU, sigmoid, tanh) introduce non-linearity, allowing networks to learn non-linear relationships. Weights are updated via backpropagation during training.
+ Why you care
Neural networks are the engine of modern AI. Knowing their structure—layers, parameters, activation functions—helps you understand model capacity, training time, and inference speed. Architecture choices (depth, width, skip connections) directly impact performance.
⚙ In Jarvis
All of Jarvis's language models (Mistral, Claude, Gemini) are neural networks. The Mac Studio M4 Max with 36GB unified memory can handle inference on quantized versions of these networks efficiently. Understanding NN capacity helps you decide: can this run locally, or does it need cloud API?

Large Language Model

LLM
Foundations

A neural network trained on vast amounts of text to predict the next token in a sequence. "Large" refers to scale: billions to trillions of parameters. LLMs can answer questions, write code, translate languages, and reason about complex topics by learning patterns from diverse text data.

+ How it works
During pre-training, the model reads text token-by-token and learns to predict the next token. This simple objective drives the model to learn grammar, facts, reasoning, and world knowledge. At inference, you feed it a prompt, and it generates responses by sampling or greedy-selecting tokens sequentially.
+ Why you care
LLMs are the primary interface to modern AI. You need to understand: prompt engineering, context windows, token budgets, output variance (temperature), and cost. Every LLM-powered feature depends on these properties.
⚙ In Jarvis
Jarvis routes LLM queries: Mistral Small 3.1 for chat/routing ($0, local), Gemini 2.0 Flash for n8n synthesis ($0.40/1M tokens via OpenRouter), Claude Sonnet 4 for reasoning ($3/1M via Anthropic). Each has different context windows and capabilities—understanding these differences drives routing logic.

Foundation Model

FM
Foundations

A large-scale model pre-trained on broad, unlabeled data that can be adapted (fine-tuned) for many downstream tasks. Foundation models include LLMs (text), vision transformers (images), and multimodal models. They form the "foundation" for specialized applications.

+ How it works
Foundation models are trained on massive, diverse datasets via self-supervised learning (e.g., next-token prediction). This produces a general-purpose model that captures broad knowledge. You then fine-tune on task-specific data or use in-context learning (prompting) to adapt it.
+ Why you care
Foundation models democratize AI—you don't train from scratch. The tradeoff: you're locked into the model's capabilities and biases. Knowing which foundation model fits your task (size, cost, latency, knowledge cutoff) is critical.
⚙ In Jarvis
All models Jarvis uses are foundation models: Mistral Small 3.1 (OSS, local), Claude Sonnet 4 (Anthropic), Gemini 2.0 Flash (Google). Jarvis doesn't fine-tune—it composes these FMs via prompt engineering and routing. This is the modern "app stack": orchestrate FMs rather than train them.

Generative AI

GenAI
Foundations

AI systems designed to generate new content—text, images, audio, code—rather than just classify or retrieve. GenAI models learn the distribution of training data and sample from it to create novel, original outputs conditioned on prompts or context.

+ How it works
Generative models learn to represent data distributions. LLMs predict next tokens; diffusion models denoise images; GANs have generators and discriminators. The key: instead of mapping input → label, they learn to produce diverse, coherent outputs that match the data distribution.
+ Why you care
Generative models are driving the current AI boom. Understanding their probabilistic nature—why they sometimes hallucinate, why outputs vary, why temperature matters—is essential for building reliable systems. Garbage input (or prompt) → garbage output.
⚙ In Jarvis
Jarvis's core is generative: Claude and Mistral generate text responses, Gemini Flash generates n8n workflow JSON. Jarvis manages temperature (~0.7 for balanced creativity/consistency) and uses system prompts to shape generation. Hallucinations are real—Jarvis validates outputs via type-checking and webhook returns.

Natural Language Processing

NLP
Foundations

The field of AI focused on enabling computers to understand, interpret, and generate human language. NLP tasks include sentiment analysis, machine translation, question answering, named entity recognition, and text summarization.

+ How it works
NLP systems tokenize text, embed tokens into vector spaces, and apply neural networks (transformers, RNNs) to learn linguistic patterns. Modern NLP is dominated by transformer-based language models that attend to context globally rather than processing sequentially.
+ Why you care
Every chatbot, email classifier, search engine, and code generator is built on NLP. Understanding tokenization, embeddings, and context windows helps you debug why a model gives unexpected outputs or runs out of context.
⚙ In Jarvis
NLP powers Jarvis's entire interaction layer: message parsing in Discord (OpenClaw, Hermes), email understanding in Pipeline Intelligence, and task classification in Google Tasks Kanban. The AM Brief and PM Recap workflows rely on NLP to extract intent from unstructured input and route it to the right model.

Multimodal

MM
Foundations

AI models that process and reason across multiple data modalities simultaneously—text, images, audio, video. Multimodal models learn shared representations, enabling capabilities like image captioning, visual question answering, and cross-modal search.

+ How it works
Multimodal systems encode different modalities into a shared embedding space via separate encoders (vision encoder, text encoder). A fusion mechanism (cross-attention, concatenation, or gated fusion) combines representations. The model then learns joint distributions across modalities.
+ Why you care
Multimodal models unlock new use cases: analyzing screenshots, captioning images, understanding diagrams. They're more complex (larger models, longer inference) but far more capable. Cost and latency trade-offs are critical.
⚙ In Jarvis
Gemini 2.0 Flash and Claude Sonnet are multimodal—they accept images. Jarvis doesn't currently use image input heavily, but it could: imagine analyzing screenshots of email for automated triage, or parsing OCR'd documents. Multimodal capability is there; use case optimization pending.

Token / Tokenization

TOKEN
Foundations

A token is the smallest unit of text that a language model processes—usually a word, subword, or character. Tokenization is the process of breaking text into tokens using a vocabulary (e.g., BPE, WordPiece). Different models use different tokenizers.

+ How it works
Byte-Pair Encoding (BPE) is common: start with character-level tokens, iteratively merge frequent pairs until reaching vocab size. A token ID is then mapped to an embedding vector. The tokenizer must be inverted at output (token IDs → text).
+ Why you care
Tokens = cost and latency. A 1000-character prompt might be 250 tokens (cheap) or 500 tokens (expensive) depending on tokenization. You pay per token. Understanding token counts helps estimate API costs and context window usage.
⚙ In Jarvis
Gemini Flash costs $0.40/1M input tokens; Claude Sonnet ~$3/1M. Jarvis batches requests to maximize token efficiency. A typical AM Brief might consume 10K tokens (cheap). But analyzing 120 days of email for Pipeline Intelligence uses many tokens—Jarvis batches this as a background task to avoid real-time latency.

Context Window

CTX
Foundations

The maximum number of tokens a model can "see" at once—the combined length of prompt + generated output. Larger context windows allow processing longer documents, multi-turn conversations, or more in-context examples. Typical range: 4K to 200K+ tokens.

+ How it works
During inference, the model attends to all tokens in the context window via self-attention. Larger context requires more memory and computation (attention is O(n²) in context length). Long context enables RAG (retrieval-augmented generation): stuff relevant documents into the prompt.
+ Why you care
Context window limits force tradeoffs: Can you include the full document, or must you summarize? Can you keep full conversation history, or truncate old messages? Knowing model context sizes helps you design interactions.
⚙ In Jarvis
Mistral Small has 32K context; Claude Sonnet has 200K; Gemini Flash has 1M tokens. Jarvis Pipeline Intelligence leverages Gemini's long context: it can ingest 120 days of email (hundreds of thousands of tokens) in one shot, classify accounts, and return results. Without long context, this would require multiple API calls or summarization.

Embedding

EMB
Foundations

A vector representation of text, images, or other data in a continuous, high-dimensional space. Embeddings capture semantic meaning: similar concepts have similar embeddings. They enable similarity search, clustering, and semantic retrieval without exact keyword matching.

+ How it works
A model (transformer, CNN, etc.) encodes input into a fixed-size vector. The model is trained such that semantically similar inputs produce similar embeddings. Cosine similarity or Euclidean distance in embedding space measures semantic closeness.
+ Why you care
Embeddings enable semantic search: find documents similar to a query without keyword matching. They're the foundation of RAG systems. Understanding embedding quality, dimensionality, and model choice (OpenAI, open-source) drives search relevance.
⚙ In Jarvis
Jarvis's Obsidian vault could benefit from embedding-based semantic search: index all notes, embed queries, find contextually relevant notes. Currently Jarvis uses exact keyword search. Adding embeddings (via local all-MiniLM or Ollama embeddings endpoint) would enable richer retrieval for the AM Brief and reasoning workflows.

Prompt / Prompt Engineering

PROMPT
Foundations

A prompt is the input text you give an LLM. Prompt engineering is the art of crafting prompts to elicit desired outputs. Techniques include examples (few-shot learning), explicit instructions, role-playing, and chain-of-thought reasoning.

+ How it works
LLMs are few-shot learners—they learn from examples in the prompt. Prompt engineering exploits this: structure prompts with examples, instructions, and constraints. The model then patterns-matches and generates similar outputs. Clear, specific prompts yield better results than vague ones.
+ Why you care
Prompt engineering is your primary tool for steering LLM behavior without training. Good prompts can 10x model quality. Understanding techniques (chain-of-thought, role prompting, structured formats) is critical for reliable outputs.
⚙ In Jarvis
Jarvis's entire intelligence layer is prompt engineering. Examples: the AM Brief prompt instructs Claude to scan emails and extract urgent items; the Pipeline Intel prompt tells Gemini to classify accounts; Discord bot prompts route messages to appropriate models. No fine-tuning—pure prompt craft. System prompts set role and constraints; user prompts provide context.

System Prompt

SYSPROMPT
Foundations

A special prompt (usually not visible to end-users) that sets the tone, role, and constraints for an LLM before the user's query. System prompts shape personality, enforce output format, and define the model's "character."

+ How it works
System prompts are prepended to the conversation context before user messages. They establish role ("You are a Python expert"), constraints ("Never write unsafe code"), and format ("Return JSON"). Models strongly condition on system prompts, making them powerful steering mechanisms.
+ Why you care
System prompts are your guardrails. Well-designed system prompts enforce safety, consistency, and output format. They're the easiest way to customize model behavior without retraining.
⚙ In Jarvis
Jarvis uses distinct system prompts per workflow: the AM Brief prompt tells Claude "You are a personal AI assistant summarizing emails"; the Pipeline Intel prompt tells Gemini "Classify sales accounts and return JSON"; Discord bot prompts route to appropriate models. System prompts are the "constitution" of each Jarvis behavior.

Temperature / Top-P / Top-K

TEMP
Foundations

Hyperparameters controlling the randomness of LLM outputs. Temperature (0-2) scales logits: low temp (0.1) = deterministic/repetitive; high temp (1.5) = creative/diverse. Top-P (nucleus sampling) and Top-K (take K most likely tokens) are alternative sampling methods.

+ How it works
Temperature: logits = logits / temp. Top-P: sample from smallest set of tokens whose cumulative probability >= P (e.g., 0.9). Top-K: sample from K highest-probability tokens. All methods shape the probability distribution before sampling the next token.
+ Why you care
These settings control quality vs. diversity tradeoff. Summarization? Low temp (0.3). Creative writing? High temp (1.0). Understanding these lets you tune outputs without model changes.
⚙ In Jarvis
Jarvis sets temperature ~0.7 for balanced outputs: creative enough for good summaries, consistent enough for JSON generation. Pipeline Intelligence uses lower temp (~0.3) to ensure deterministic account classifications. Workflow synthesis (Gemini) uses 0.8 for creative prompt suggestions.

Inference

INF
Foundations

The process of running a trained model on new data to generate outputs. Inference is distinct from training: no weight updates, just forward passes. Inference speed (latency) and throughput are critical for real-time applications.

+ How it works
Input → forward pass through learned weights → output. For LLMs, inference is autoregressive: predict token 1, append to context, predict token 2, etc. Each token generation is one forward pass (expensive). Batch processing multiple requests improves throughput.
+ Why you care
Inference cost dominates operating LLMs at scale. Latency (time to first token) and throughput (tokens/second) determine user experience and infrastructure costs. Optimizations like batching, quantization, and KV caching are essential.
⚙ In Jarvis
Mistral inference runs locally via Ollama on Mac Studio (36GB memory, M4 Max). Achieves ~20 tokens/sec. Cloud inference (Claude, Gemini) is managed by APIs—Jarvis doesn't optimize lower-level inference, but understands cost tradeoffs: local = free + latency, cloud = paid + speed.

Parameters (Model Size)

PARAMS
Foundations

The learnable weights in a neural network. Model size is measured in parameters (B = billions). Larger models (70B, 405B) have higher capacity but require more memory and compute. Smaller models (7B, 8B) are faster and cheaper but less capable.

+ How it works
Each layer in a neural network has parameters. A 24B model has 24 billion learnable values. During inference, every forward pass reads all parameters into memory. Memory usage ≈ 2 × parameters (in bytes, if using float16).
+ Why you care
Model size determines hardware requirements, latency, and quality. Larger models are smarter but slower/more expensive. You must match model size to your constraints (GPU memory, cost budget, latency requirements).
⚙ In Jarvis
Mistral Small 3.1 = 24B params, fits in Mac Studio's 36GB memory via quantization. Claude Sonnet is much larger (300B+, estimate) but hidden in API. Gemini Flash is smaller/faster (~40B estimate). Jarvis chooses model size by task: Mistral for fast local chat, Claude for deep reasoning, Gemini for synthesis.

Weights & Biases

W&B
Foundations

Weights are the learned parameters that connect neurons; biases are per-neuron offsets. Together, they encode a neural network's knowledge. Weights are the "memory" of what the model learned during training; biases allow shifting the activation function.

+ How it works
Each neuron computes: output = activation(Σ(weight × input) + bias). Weights scale inputs; biases shift the result. During training, backpropagation updates weights and biases to minimize loss. At inference, these learned values are frozen.
+ Why you care
Weights encode what the model learned. Inspecting weight distributions can reveal overfitting. Quantizing weights (reducing precision) saves memory at the cost of accuracy. Understanding weight structure helps you debug model behavior.
⚙ In Jarvis
Mistral Small's weights (24B params) are stored in GGUF format, quantized to Q4_K_M (4-bit) to fit in 36GB memory. This reduces model size from ~48GB (float32) to ~15GB. Quantization reduces weight precision but preserves output quality for generation tasks. Jarvis doesn't inspect weights—it trusts the foundation models.

Layer Normalization

NORM
Architectures

A normalization technique that standardizes activations across features within each token, stabilizing training. Layer norm rescales outputs to mean 0, variance 1, then applies learnable scale/shift. Applied before or after each sub-layer in transformers.

+ How it works
For each token: compute mean and variance across features, subtract mean, divide by std, scale by gamma + beta (learned parameters). Unlike batch norm (normalizes across batch), layer norm works per-token, making it stable in generative tasks where batch size varies.
+ Why you care
Layer norm is critical for training stability. Without it, transformers are hard to train. Understanding normalization helps debug training issues (gradient flow, activation scales).
⚙ In Jarvis
Mistral's transformer uses layer norm between blocks. Jarvis doesn't tune normalization—it's baked into the pre-trained model. But understanding that norm stabilizes training explains why you can't just stack random layers; careful architecture design is required.

Residual Connections

SKIP
Architectures

Skip connections (x → x + f(x)) that allow gradients to flow directly through deep networks, enabling training of hundreds of layers. Without residual connections, gradients vanish in very deep networks. Also improves information flow in forward pass.

+ How it works
Input x → sub-layer (attention, FFN) → output f(x) → add back x → output + residual. This allows each layer to learn residuals (differences) rather than absolute functions. Gradient can backprop directly through skip paths, avoiding vanishing gradients.
+ Why you care
Residual connections enable deep models. Modern transformers have 100+ layers, only possible because of skip connections. Understanding this explains why depth doesn't automatically improve quality—vanishing gradients can still occur.
⚙ In Jarvis
Mistral Small uses residual connections throughout its 24B parameter structure. This is standard in modern transformers. Jarvis benefits from these design choices—deep models enable complex reasoning—without needing to understand internal details.

Mixture of Experts (MoE)

MOE
Architectures

An architecture where FFN layers are replaced with multiple expert networks. A gating function routes each token to select experts, activating only a fraction of parameters per token. This allows scaling parameter count without proportional compute increase.

+ How it works
For each token, compute gate scores (softmax over expert indices). Select top-K experts (e.g., 2-4). Route token through selected experts, weighted by gate scores. Experts are specialized FFN networks learned during training. Dynamic routing allows different tokens to use different experts.
+ Why you care
MoE enables "pretend" large models: a model with 300B total parameters might activate only 100B per token (smaller compute footprint). Useful for scaling with efficiency constraints. Downside: more complex training and inference.
⚙ In Jarvis
Some large models (possibly Gemini, unknown architecture) use MoE. Jarvis doesn't expose MoE details, but benefits from efficiency: Gemini Flash is fast partly because it selectively activates experts, not running full dense computation.

Diffusion Models

DIFF
Architectures

Generative models that learn to reverse noise. Training corrupts data with Gaussian noise progressively; the model learns to denoise. At inference, start from random noise and iteratively denoise to generate samples. Powers DALL-E, Stable Diffusion, etc.

+ How it works
Forward process: gradually add noise to data over T timesteps. Reverse process: model learns to predict noise at each step. Training minimizes MSE between predicted and actual noise. Inference: start with noise, apply learned denoising T times. Conditioning on text enables text-to-image generation.
+ Why you care
Diffusion is the dominant generative model for images. Understanding the forward/reverse process helps reason about generation quality (more steps = better quality but slower). Latent diffusion (operating in compressed space) makes generation faster.
⚙ In Jarvis
Jarvis doesn't currently use image generation, but Gemini Flash can handle images. If Jarvis needed to generate images (e.g., creating diagrams for reports), it would likely use a diffusion-based API. Diffusion is powerful but slower than text generation.

GAN (Generative Adversarial Network)

GAN
Architectures

An architecture with two networks: Generator (creates fake samples) and Discriminator (classifies real vs. fake). They compete in adversarial training: generator improves to fool discriminator; discriminator improves to detect fakes. Trains without explicit likelihood, enabling sharp sample generation.

+ How it works
Generator: noise → sample. Discriminator: sample → real/fake. Adversarial loss: generator wants high discriminator error; discriminator wants low error. Iteratively update both. Often unstable (mode collapse, training divergence), so many tricks needed (spectral norm, progressive growing, etc.).
+ Why you care
GANs were revolutionary for image generation (2014-2019) but are less common now (diffusion models are easier to train). Understanding GANs explains adversarial robustness concerns and why some models are vulnerable to adversarial examples.
⚙ In Jarvis
Jarvis doesn't use GANs directly. Diffusion and transformer-based text models are more common now. But understanding GAN adversariness is relevant: Jarvis should be robust to adversarial prompts (e.g., prompt injection attacks).

VAE (Variational Autoencoder)

VAE
Architectures

A generative model that learns to compress data into a latent space and reconstruct it. Encoder: data → latent distribution (mean, variance). Decoder: sample from latent → reconstructed data. VAEs enable both generation (sample latent, decode) and compression.

+ How it works
Encoder outputs (μ, log σ²). During training, sample z ~ N(μ, σ²). Decoder reconstructs from z. Loss = reconstruction loss + KL divergence (regularizer forcing latent toward N(0,1)). Inference: sample from N(0,1), decode to generate new samples.
+ Why you care
VAEs are interpretable (latent space is continuous, smooth) but produce blurry samples compared to diffusion. Useful for compression, anomaly detection, and learning interpretable representations.
⚙ In Jarvis
Jarvis uses embeddings (similar idea: compressed representations), but not VAEs directly. If Jarvis implemented semantic search, VAEs could learn compact document representations. Currently simpler embedding models suffice.

State Space Models (Mamba)

SSM
Architectures

A recent alternative to transformers inspired by signal processing and differential equations. State space models process sequences with O(n) memory and O(n) computation, unlike transformer attention (O(n²)). Mamba is a prominent SSM architecture showing promise for very long sequences.

+ How it works
Reformulate sequence processing as continuous state evolution: dx/dt = Ax + Bu. Discretize and solve as a linear recurrence. Learn A (dynamics), B (input projection), C (output projection). Process tokens sequentially but avoid explicit attention (all-pairs computation).
+ Why you care
SSMs promise linear-time processing, enabling million-token contexts efficiently. Might replace transformers for very long sequences. Early results are promising, but ecosystem still developing.
⚙ In Jarvis
Gemini Flash might use SSM components (speculative). Jarvis doesn't depend on SSM vs. transformer—it uses whatever APIs expose. But understanding SSM helps explain Gemini's 1M token context: it's likely using linear-time architectures, not pure transformer attention.

Vision Transformer (ViT)

VIT
Architectures

Applying the transformer architecture to images: divide image into patches, embed patches as tokens, apply standard transformer. ViT achieves competitive results with CNNs, enabling unified architecture for vision and language (multimodal models).

+ How it works
Image (H×W×3) → patches (n × d). Add learnable cls token, positional encodings, apply transformer. Class token output → classification head. Key insight: image patches are like text tokens. Same architecture, different input modality.
+ Why you care
ViT unifies vision and language with transformers. Enables multimodal models (e.g., CLIP). Understanding patch-based processing helps reason about vision tasks and why large images are expensive (more patches = more tokens).
⚙ In Jarvis
Claude Sonnet and Gemini Flash likely use ViT for image processing. Jarvis doesn't process images heavily yet, but multimodal capability is there (e.g., analyzing screenshots of emails, OCR'd documents). ViT is likely how they encode images into shared embedding space.

🏗️ Architectures & How Models Are Built

Transformer

TFM
Architectures

The neural network architecture that powers modern LLMs, vision models, and multimodal systems. Transformers use self-attention to process inputs in parallel, enabling efficient training on massive datasets. Introduced by Vaswani et al. (2017): "Attention is All You Need."

+ How it works
Transformer stacks encoder/decoder blocks. Each block has: multi-head self-attention (allows tokens to attend to all other tokens), feed-forward networks, and residual connections. Self-attention computes query-key-value products to determine which tokens to attend to. Positional encodings preserve token order (tokens are processed in parallel).
+ Why you care
Transformers are the foundation of all modern LLMs. Understanding attention helps you reason about context—why large context windows are hard, why long documents might get lost, why token order matters less than in RNNs.
⚙ In Jarvis
Mistral, Claude, and Gemini are all transformer-based. Jarvis doesn't need to understand transformer internals to use them, but knowing that attention is O(n²) in context length helps explain why processing 120 days of email (millions of tokens) with a single model is expensive.

Self-Attention / Multi-Head Attention

ATTN
Architectures

A mechanism that allows each token to "attend to" (weight) all other tokens in the input. Multi-head attention runs this mechanism in parallel with multiple "heads," each focusing on different semantic relationships. Attention weights are learned dynamically per input.

+ How it works
For each position, compute: Query (Q), Key (K), Value (V) projections. Attention scores = softmax(Q·K^T / sqrt(d)). Output = weighted sum of Values. Multi-head: repeat with different Q,K,V projections, concatenate results. This allows learning multiple relationship types simultaneously.
+ Why you care
Attention is why transformers work: they can learn long-range dependencies. But attention is expensive (quadratic in sequence length). Understanding this explains context window limits and why very long documents are hard.
⚙ In Jarvis
Gemini Flash handles 1M token context via efficient attention mechanisms (possibly sparse or linear attention). Jarvis leverages this: Pipeline Intelligence can process 120 days of email at once. Without multi-head attention's long-range learning, you'd need hierarchical summarization or multiple API calls.

Encoder-Decoder

ENC-DEC
Architectures

A two-part architecture: Encoder processes input and creates context representations. Decoder generates output token-by-token, attending to encoder outputs. Used for sequence-to-sequence tasks: machine translation, summarization, question answering.

+ How it works
Encoder: stacked transformer blocks process entire input in parallel. Decoder: stacked blocks generate output autoregressively (one token at a time), with cross-attention to encoder outputs. Cross-attention lets decoder "focus on" relevant input parts when generating each output token.
+ Why you care
Encoder-decoder excels at structured tasks where you transform long input into long output. It's less common now (decoder-only models dominate), but understanding the pattern helps with task design: is this encode-then-generate, or just generate from context?
⚙ In Jarvis
Jarvis uses decoder-only models (Mistral, Claude, Gemini). Pipeline Intelligence encodes 120 days of email context in the prompt (implicit encoding), then Gemini decodes classifications. The boundary between encoder and decoder is blurred in modern systems—prompt = encoder, generation = decoder.

Decoder-Only (GPT-style)

DEC-ONLY
Architectures

A simplified transformer with only the decoder stack (no separate encoder). All inputs are concatenated, and the model predicts next tokens autoregressively. GPT, Mistral, Claude all use this architecture. Simpler than encoder-decoder but still powerful.

+ How it works
Input (prompt + context) → transformer decoder blocks → output (next token). Causal masking ensures tokens only attend to previous tokens (can't cheat by looking ahead). Autoregressive generation: input → token 1, then [input + token 1] → token 2, etc.
+ Why you care
Decoder-only is simpler and more flexible than encoder-decoder. You can prompt them with anything (context, instructions, examples) and they generalize. Most modern LLMs are decoder-only because they're easier to scale and more versatile.
⚙ In Jarvis
All of Jarvis's models are decoder-only: Mistral Small 3.1, Claude Sonnet, Gemini Flash. This is why prompt engineering works—you feed context + instructions, and the decoder generates responses. No separate encoding step; the prompt is the context.

Feed-Forward Network

FFN
Architectures

A simple neural network (typically 2 dense layers with a non-linear activation) applied independently to each token's representation. Feed-forward layers (also called position-wise FFN) transform representations without mixing across tokens. Interleaved with attention in transformer blocks.

+ How it works
After self-attention (which mixes information across tokens), FFN applies: x' = W2 * activation(W1 * x). Typical activation: ReLU or GELU. This nonlinear transformation allows learning complex functions per-token. FFN typically expands hidden dimension (e.g., 4x), then projects back.
+ Why you care
FFN is where much of the model's "knowledge" (facts, reasoning) is encoded. Understanding FFN structure helps explain why models can memorize training data and why distillation works.
⚙ In Jarvis
Mistral and other transformers interleave attention + FFN blocks. FFN is where facts about the world are stored. When Jarvis prompts Claude about a current event, Claude's FFN contains that knowledge (if trained on recent data). Jarvis doesn't fine-tune FFN—it relies on pre-trained knowledge.

Positional Encoding

POS-ENC
Architectures

A technique to inject token position information into transformer inputs. Since transformers process all tokens in parallel (unlike RNNs), they don't inherently know "token order." Positional encodings (sinusoidal functions or learned embeddings) add position-aware signals.

+ How it works
Sinusoidal encoding: PE(pos, i) = sin(pos/10000^(2i/d)) for even i, cos(pos/10000^(2i/d)) for odd i. This creates unique, smooth patterns for each position, allowing attention to respect token order. Modern variants: learned absolute/relative positional embeddings.
+ Why you care
Positional encodings determine context window size (longer sequences need more encoding range). Understanding this helps explain why extending context requires retraining or new positional encoding strategies.
⚙ In Jarvis
Mistral's 32K context and Claude's 200K context are partly limited by positional encoding design. Gemini's 1M context uses advanced positional encoding (likely ALiBi or rotary embeddings). Jarvis doesn't need to tune positional encodings—that's pre-trained—but the distinction explains why context windows vary by model.

Layer Normalization

NORM
Architectures

A normalization technique that standardizes activations across features within each token, stabilizing training. Layer norm rescales outputs to mean 0, variance 1, then applies learnable scale/shift. Applied before or after each sub-layer in transformers.

+ How it works
For each token: compute mean and variance across features, subtract mean, divide by std, scale by gamma + beta (learned parameters). Unlike batch norm (normalizes across batch), layer norm works per-token, making it stable in generative tasks where batch size varies.
+ Why you care
Layer norm is critical for training stability. Without it, transformers are hard to train. Understanding normalization helps debug training issues (gradient flow, activation scales).
⚙ In Jarvis
Mistral's transformer uses layer norm between blocks. Jarvis doesn't tune normalization—it's baked into the pre-trained model. But understanding that norm stabilizes training explains why you can't just stack random layers; careful architecture design is required.

Residual Connections

SKIP
Architectures

Skip connections (x → x + f(x)) that allow gradients to flow directly through deep networks, enabling training of hundreds of layers. Without residual connections, gradients vanish in very deep networks. Also improves information flow in forward pass.

+ How it works
Input x → sub-layer (attention, FFN) → output f(x) → add back x → output + residual. This allows each layer to learn residuals (differences) rather than absolute functions. Gradient can backprop directly through skip paths, avoiding vanishing gradients.
+ Why you care
Residual connections enable deep models. Modern transformers have 100+ layers, only possible because of skip connections. Understanding this explains why depth doesn't automatically improve quality—vanishing gradients can still occur.
⚙ In Jarvis
Mistral Small uses residual connections throughout its 24B parameter structure. This is standard in modern transformers. Jarvis benefits from these design choices—deep models enable complex reasoning—without needing to understand internal details.

Mixture of Experts (MoE)

MOE
Architectures

An architecture where FFN layers are replaced with multiple expert networks. A gating function routes each token to select experts, activating only a fraction of parameters per token. This allows scaling parameter count without proportional compute increase.

+ How it works
For each token, compute gate scores (softmax over expert indices). Select top-K experts (e.g., 2-4). Route token through selected experts, weighted by gate scores. Experts are specialized FFN networks learned during training. Dynamic routing allows different tokens to use different experts.
+ Why you care
MoE enables "pretend" large models: a model with 300B total parameters might activate only 100B per token (smaller compute footprint). Useful for scaling with efficiency constraints. Downside: more complex training and inference.
⚙ In Jarvis
Some large models (possibly Gemini, unknown architecture) use MoE. Jarvis doesn't expose MoE details, but benefits from efficiency: Gemini Flash is fast partly because it selectively activates experts, not running full dense computation.

Diffusion Models

DIFF
Architectures

Generative models that learn to reverse noise. Training corrupts data with Gaussian noise progressively; the model learns to denoise. At inference, start from random noise and iteratively denoise to generate samples. Powers DALL-E, Stable Diffusion, etc.

+ How it works
Forward process: gradually add noise to data over T timesteps. Reverse process: model learns to predict noise at each step. Training minimizes MSE between predicted and actual noise. Inference: start with noise, apply learned denoising T times. Conditioning on text enables text-to-image generation.
+ Why you care
Diffusion is the dominant generative model for images. Understanding the forward/reverse process helps reason about generation quality (more steps = better quality but slower). Latent diffusion (operating in compressed space) makes generation faster.
⚙ In Jarvis
Jarvis doesn't currently use image generation, but Gemini Flash can handle images. If Jarvis needed to generate images (e.g., creating diagrams for reports), it would likely use a diffusion-based API. Diffusion is powerful but slower than text generation.

GAN (Generative Adversarial Network)

GAN
Architectures

An architecture with two networks: Generator (creates fake samples) and Discriminator (classifies real vs. fake). They compete in adversarial training: generator improves to fool discriminator; discriminator improves to detect fakes. Trains without explicit likelihood, enabling sharp sample generation.

+ How it works
Generator: noise → sample. Discriminator: sample → real/fake. Adversarial loss: generator wants high discriminator error; discriminator wants low error. Iteratively update both. Often unstable (mode collapse, training divergence), so many tricks needed (spectral norm, progressive growing, etc.).
+ Why you care
GANs were revolutionary for image generation (2014-2019) but are less common now (diffusion models are easier to train). Understanding GANs explains adversarial robustness concerns and why some models are vulnerable to adversarial examples.
⚙ In Jarvis
Jarvis doesn't use GANs directly. Diffusion and transformer-based text models are more common now. But understanding GAN adversariness is relevant: Jarvis should be robust to adversarial prompts (e.g., prompt injection attacks).

VAE (Variational Autoencoder)

VAE
Architectures

A generative model that learns to compress data into a latent space and reconstruct it. Encoder: data → latent distribution (mean, variance). Decoder: sample from latent → reconstructed data. VAEs enable both generation (sample latent, decode) and compression.

+ How it works
Encoder outputs (μ, log σ²). During training, sample z ~ N(μ, σ²). Decoder reconstructs from z. Loss = reconstruction loss + KL divergence (regularizer forcing latent toward N(0,1)). Inference: sample from N(0,1), decode to generate new samples.
+ Why you care
VAEs are interpretable (latent space is continuous, smooth) but produce blurry samples compared to diffusion. Useful for compression, anomaly detection, and learning interpretable representations.
⚙ In Jarvis
Jarvis uses embeddings (similar idea: compressed representations), but not VAEs directly. If Jarvis implemented semantic search, VAEs could learn compact document representations. Currently simpler embedding models suffice.

State Space Models (Mamba)

SSM
Architectures

A recent alternative to transformers inspired by signal processing and differential equations. State space models process sequences with O(n) memory and O(n) computation, unlike transformer attention (O(n²)). Mamba is a prominent SSM architecture showing promise for very long sequences.

+ How it works
Reformulate sequence processing as continuous state evolution: dx/dt = Ax + Bu. Discretize and solve as a linear recurrence. Learn A (dynamics), B (input projection), C (output projection). Process tokens sequentially but avoid explicit attention (all-pairs computation).
+ Why you care
SSMs promise linear-time processing, enabling million-token contexts efficiently. Might replace transformers for very long sequences. Early results are promising, but ecosystem still developing.
⚙ In Jarvis
Gemini Flash might use SSM components (speculative). Jarvis doesn't depend on SSM vs. transformer—it uses whatever APIs expose. But understanding SSM helps explain Gemini's 1M token context: it's likely using linear-time architectures, not pure transformer attention.

Vision Transformer (ViT)

VIT
Architectures

Applying the transformer architecture to images: divide image into patches, embed patches as tokens, apply standard transformer. ViT achieves competitive results with CNNs, enabling unified architecture for vision and language (multimodal models).

+ How it works
Image (H×W×3) → patches (n × d). Add learnable cls token, positional encodings, apply transformer. Class token output → classification head. Key insight: image patches are like text tokens. Same architecture, different input modality.
+ Why you care
ViT unifies vision and language with transformers. Enables multimodal models (e.g., CLIP). Understanding patch-based processing helps reason about vision tasks and why large images are expensive (more patches = more tokens).
⚙ In Jarvis
Claude Sonnet and Gemini Flash likely use ViT for image processing. Jarvis doesn't process images heavily yet, but multimodal capability is there (e.g., analyzing screenshots of emails, OCR'd documents). ViT is likely how they encode images into shared embedding space.

📚 Training & Learning

Pre-training

PT
Training

The initial training phase where a model learns from massive unlabeled data. LLMs are pre-trained on next-token prediction using trillions of tokens. This phase encodes broad world knowledge, language patterns, and reasoning abilities.

+ How it works
Feed data → compute loss (e.g., cross-entropy on next token) → backprop → update weights. Repeat over trillions of tokens, using GPUs/TPUs. Learning rate schedules, gradient clipping, and checkpointing manage stability. Pre-training is expensive (weeks/months, millions of dollars).
+ Why you care
Pre-training is where model quality is determined. Larger models, better data, longer training = better pre-trained models. You don't pre-train yourself (costs prohibitive), but understanding pre-training explains why foundation models are valuable.
⚙ In Jarvis
All Jarvis models are pre-trained by others: Mistral (by Mistral AI), Claude (by Anthropic), Gemini (by Google). Jarvis doesn't do pre-training—it uses these pre-trained models as foundations and adapts via prompt engineering or fine-tuning.

Fine-tuning

FT
Training

Continued training of a pre-trained model on task-specific data. Fine-tuning adapts the model to perform well on your specific task (classification, QA, code generation) without retraining from scratch.

+ How it works
Start with pre-trained weights. Feed task-specific data, apply same training procedure but with lower learning rate (to avoid catastrophic forgetting). After 1-10 epochs, the model learns task patterns while retaining pre-training knowledge.
+ Why you care
Fine-tuning is how you adapt pre-trained models to your domain. Cheap (hours, not weeks) and effective. Trade-off: requires labeled data. LoRA/QLoRA make fine-tuning feasible on consumer hardware.
⚙ In Jarvis
Jarvis doesn't fine-tune its models. Instead, it uses prompt engineering (few-shot examples, system prompts) to adapt models. Fine-tuning would be useful for: making models understand Jarvis-specific terminology, or learning Jarvis's decision-making patterns. Not implemented, but possible future optimization.

⚡ Inference & Running Models

Inference (detailed)

INF
Inference

Running a trained model on new data to generate outputs. For LLMs, inference is autoregressive token generation: predict token 1, append to context, predict token 2, repeat. Inference is fundamentally different from training (no weight updates, just forward passes).

+ How it works
Input prompt → tokenize → embed → forward pass through all layers → logits → sample/greedy → token ID → detokenize → output text. Repeat, appending each token to context. For efficiency, use KV caching: cache key/value tensors from previous tokens.
+ Why you care
Inference cost is the limiting factor in production. Understanding bottlenecks (latency, memory, throughput) helps optimize. KV caching, batching, and quantization are critical for cost-effective inference.
⚙ In Jarvis
Mistral runs locally: Ollama handles inference server, tokenization, KV caching, batching. Claude and Gemini are remote: Jarvis sends prompt, API handles inference, returns output. Jarvis orchestrates routing: simple queries → Mistral (free), complex reasoning → Claude (paid).

Quantization (INT4, INT8, GPTQ, AWQ, GGUF)

QUANT
Inference

Reducing model precision (float32 → float16 → int8 → int4) to decrease memory and compute requirements. Quantized models are smaller, faster, and fit on consumer hardware. Trade-off: slight accuracy loss, but usually acceptable for generation.

+ How it works
Quantization: map float weights to lower-precision integers. INT8: 1 byte/weight (8x smaller than float32). INT4: 4 bits/weight (16x smaller). Dequantization: convert back to float during computation. Methods: post-training quantization (simple), fine-tuned quantization (better quality).
+ Why you care
Quantization is how large models run on consumer GPUs/CPUs. GGUF (for llama.cpp), GPTQ, AWQ are popular formats. Understanding quantization helps you choose: do you need accuracy (int8) or speed (int4)?
⚙ In Jarvis
Mistral Small runs as GGUF Q4_K_M (4-bit quantization). This reduces ~48GB (float32) to ~15GB, fitting in Mac Studio's 36GB memory with headroom. Quality is excellent for chat/generation. GGUF also enables CPU inference (no GPU needed). Trade-off: slightly slower than FP32 but acceptable (~20 tokens/sec).

Section 5: Agents, Memory & Orchestration

Autonomous systems that reason, remember, and coordinate work across tools and services.

AI Agent — Agent
An autonomous system that perceives its environment, reasons about goals, and takes actions to achieve them—without explicit step-by-step instructions.
How it works: Observes state → reasons about next step → executes action → observes new state → repeats. Unlike a chatbot that responds once, agents loop autonomously, using tools and memory to achieve complex goals.
Why you care: Agents let you delegate multi-step problems and automate entire workflows. Instead of "write an email," you say "reach out to inactive customers and propose a call."
In Jarvis

Jarvis runs as a distributed agent system: OpenClaw routes requests, Hermes executes skills. They communicate via Discord webhooks and task queues, each maintaining independent state and memory.

Tool Use / Function Calling — Agent
The ability for an LLM to request execution of external functions—APIs, code, database queries—as part of reasoning.
How it works: You define available functions with parameters. The LLM reasons "I need to fetch user data" and calls your function, you return the result, it reasons further. Enables agentic loops.
Why you care: Without tool use, LLMs can only text-chat. With it, they become workers: querying databases, calling APIs, triggering webhooks, executing scripts.
In Jarvis

Claude uses tool calls to request actions from Hermes: "list_tasks", "post_to_discord", "query_gmail". MCP standardizes these interfaces across Python, Node, and browser contexts.

ReAct Pattern Reason + Act Agent
Alternating cycle: LLM reasons about the problem, acts (calls a tool), observes the result, reasons again, acts again.
How it works: Thought → Action → Observation → Thought → Action → ... until goal reached. The reasoning step is explicit (shown in tool calls or internal monologue), making the agent's logic transparent.
Why you care: ReAct is more reliable than pure tool calling because reasoning is explicit and verifiable. You can see why the agent chose each action, making debugging easier.
In Jarvis

The AM Brief workflow: Claude reasons "I need yesterday's Gmail + calendar," acts (calls list_emails, get_calendar), observes results, synthesizes (Gemini Flash), then acts (post_to_discord, create_task).

Chain of Thought (CoT) CoT Agent
Prompting technique where the LLM explicitly writes out reasoning steps before giving an answer, improving accuracy especially on complex tasks.
How it works: Instead of "Summarize this," you say "Think step-by-step: what are the key themes? How do they relate? Now summarize." The model's verbalization of reasoning improves the final answer.
Why you care: CoT makes reasoning observable and verifiable. A model that shows its work is more trustworthy. Also, thinking aloud often improves the solution itself.
In Jarvis

Claude Sonnet is used for #thinking channel questions because extended thinking + CoT prompts produce better decisions. Mistral (faster, cheaper) skips this for casual chat.

Multi-Agent Systems — Agent
Multiple independent agents, each with distinct roles and capabilities, coordinating via communication protocols to achieve shared or complementary goals.
How it works: Agent A (router) receives a request, determines which agents (specialist workers) should handle it, communicates with them, aggregates results, and responds. Agents may pass messages, share memory, or trigger each other.
Why you care: Multi-agent systems scale expertise. One agent is good, but a router + 5 specialists is better. Also enables resilience: if one agent crashes, others keep working.
In Jarvis

OpenClaw (router) + Hermes (executor) + Claude (reasoner) + Gemini (synthesizer). Each has distinct responsibilities. They communicate via Discord webhooks and n8n workflows, forming a loosely-coupled system.

Orchestration — Agent
Coordinating multiple tools, services, and agents in a defined sequence or conditional flow to achieve a business outcome.
How it works: Workflow engine (like n8n) defines steps: "trigger on webhook → fetch Gmail → call Claude → parse result → post to Discord → create task." Each step may have conditions (if X, then Y).
Why you care: Orchestration turns ad-hoc automation into repeatable, reliable processes. No code needed; visual workflow builders let anyone set up complex automations.
In Jarvis

n8n manages 7 core workflows: AM Brief, PM Recap, Pipeline Intelligence, Config Backup, Nightly Historian, Mutual Watchdog, Task Inbox Router. Each is a DAG (directed acyclic graph) of steps.

RAG (Retrieval Augmented Generation) RAG Agent
Augmenting an LLM's response by first retrieving relevant documents, knowledge base entries, or embeddings, then feeding them into the prompt.
How it works: Query → search vector DB or knowledge base → retrieve top-K relevant docs → insert into prompt → LLM generates answer grounded in those docs. Prevents hallucination by giving the model facts to cite.
Why you care: Without RAG, LLMs hallucinate and don't know about proprietary data. With RAG, they become knowledge workers: answering questions about your docs, vault, emails, customer data.
In Jarvis

Obsidian vault is the knowledge base. When Claude receives a question, it retrieves relevant vault entries, email history, or decision logs, then reasons over them. This grounds answers in Jarvis's actual history.

Vector Database / Vector Store VDB Agent
Database optimized for storing and searching embeddings (semantic vectors). Enables similarity search: find documents semantically similar to a query, not just keyword matches.
How it works: Documents → embed (convert to vectors) → store in VDB. Query → embed → find K nearest neighbors (cosine similarity) → return. Fast even with millions of vectors.
Why you care: Semantic search finds "documents about company strategy" even if they don't use that phrase. Keyword search (grep) is brittle; vector search understands meaning.
In Jarvis

Uses Anthropic's native embeddings and in-memory vector search for vault queries. Planned: Pinecone or Weaviate for scalable semantic search across email archives and decision logs.

Memory (short-term, long-term, episodic) — Agent
Mechanisms for agents to retain and retrieve information across sessions or turns. Short-term: active conversation. Long-term: persistent knowledge base. Episodic: specific past events/conversations.
How it works: Short-term: context window holds current conversation. Long-term: distilled knowledge written to persistent storage (files, DB). Episodic: logs of specific events with timestamps and outcomes, indexed for retrieval.
Why you care: Without memory, agents reset each session. With memory, they build context, learn from past mistakes, and avoid repeating work. Memory is what makes agents "mature" over time.
In Jarvis

Short-term: active conversation context. Long-term: MEMORY.md distilled nightly. Episodic: daily logs in ixon-daily-context.md, decision logs in DECISIONS/*.md. Mutual watchdog auto-restarts if memory process fails.

MCP (Model Context Protocol) MCP Agent
Anthropic's open standard for connecting LLMs to external tools and data sources. Standardized interface so one LLM can talk to many providers (APIs, databases, files, code).
How it works: MCP server exposes resources and tools (read_file, query_api, execute_code). MCP client (Claude, etc.) requests them via standardized JSON-RPC. Enables composable, modular tool ecosystems.
Why you care: MCP is a bridge: your tools work with any MCP-compatible LLM, and any MCP server works with any LLM. No more custom integrations for each LLM+tool pair.
In Jarvis

Hermes exposes tools via MCP: file_read, task_list, discord_post, gmail_query, calendar_fetch. Claude calls these through standard MCP client. Enables Jarvis to scale with new LLM providers without rewriting tool code.

Webhooks — Agent
HTTP callbacks: instead of polling for updates, a service sends POST requests to your URL when events occur.
How it works: You register a URL with a service. When an event happens (task created, email received, form submitted), the service POSTs data to your URL. Your app processes it immediately.
Why you care: Webhooks are real-time and efficient. No polling overhead. Systems react instantly to events, enabling reactive automation and event-driven architectures.
In Jarvis

Google Tasks webhook → n8n Task Inbox Router. Discord message → OpenClaw router. External tools POST to Hermes webhook. Webhooks make Jarvis reactive; it doesn't have to poll for changes.

Workflow Engine (n8n, Make, Zapier) — Agent
No-code/low-code platform for building automations: drag-and-drop steps, conditions, data transformations, and integrations without writing code.
How it works: UI shows available nodes (triggers, actions, conditionals). You connect them: trigger on event → call API → transform data → send notification. Workflow executes on schedule or webhook.
Why you care: Workflow engines democratize automation. Non-engineers can build complex integrations. Faster iteration than code. Visual debugging.
In Jarvis

n8n hosts 7 core workflows (AM Brief, PM Recap, Pipeline Intelligence, etc.). Self-hosted on NVMe Ubuntu box. n8n's native Google integrations + extensibility via HTTP + JavaScript make it the glue holding Jarvis together.

Task Decomposition — Agent
Splitting a complex goal into smaller, manageable subtasks that an agent or team can execute sequentially or in parallel.
How it works: Agent receives "build a report." It decomposes: fetch data → clean → analyze → format → send. Each subtask is simpler, easier to verify, and can be delegated.
Why you care: Decomposition makes complex tasks tractable. Easier to parallelize. Enables error recovery: if one subtask fails, restart just that one, not the whole task.
In Jarvis

Pipeline Intelligence decomposes as: scan Gmail (120 days) → categorize by account → calculate engagement → format summary → post Discord + update vault. Each step is independent and testable.

Planning (in agents) — Agent
Agent reasoning about a sequence of actions to reach a goal, considering constraints, dependencies, and outcomes.
How it works: Agent thinks: "To reach goal X, I need Y and Z first. Y requires calling API A and B. Z requires waiting for external event." It builds a plan, executes it step-by-step, adjusting if needed.
Why you care: Good planning avoids wasted steps and circular logic. An agent that plans ahead is more efficient and reliable. Planning also enables recovery: "step 3 failed, skip to step 4."
In Jarvis

Claude plans multi-step tasks for #thinking channel: "To answer this, I need context from vault, recent emails, and decision history. I'll fetch those, reason, then respond." Explicit planning makes reasoning transparent.

Guardrails / Output Validation — Agent
Constraints applied to agent outputs to ensure they're safe, valid, and aligned with system rules. Can be structural (schema validation) or behavioral (blacklists, approval gates).
How it works: Before executing an agent's action, validate it: Is the output valid JSON? Does it respect schema? Is the action within approved scope? If not, reject and ask the agent to fix it.
Why you care: Guardrails prevent agent failures cascading. A bad API call due to malformed data gets caught before execution. Safety rules keep agents from overstepping (e.g., deleting without approval).
In Jarvis

n8n validates outputs before execution. Hermes checks tool calls match approved schemas. Sensitive actions (delete, send email to new recipient) require approval gate. Mutual watchdog monitors for anomalies.

Section 6: Tools, Platforms & Infrastructure

The technical building blocks: APIs, CLIs, containers, orchestration, and cloud platforms that power modern AI systems.

API (Application Programming Interface) API Infrastructure
Contract for how two software systems communicate. You make requests (often HTTP), the API responds with data or performs actions. Standardized way to integrate services.
How it works: You construct a request (method, endpoint, headers, body), send it to the API server, receive a response (status code, headers, body). Different APIs have different formats (REST, GraphQL, gRPC), but the pattern is the same: structured input → processing → structured output.
Why you care: APIs are how services talk to each other. They enable modular systems: your app uses Gmail API, Stripe API, OpenAI API. Without APIs, everything would be isolated siloes.
In Jarvis

Calls dozens of APIs: Gmail (read emails), Google Calendar (fetch events), Google Tasks (CRUD), OpenRouter (LLM requests), Discord (webhooks), n8n (trigger workflows). MCP standardizes these interfaces so Claude can call any API through a common protocol.

REST API / HTTP Methods — Infrastructure
Architectural style where HTTP methods (GET, POST, PUT, DELETE) map to operations on resources. GET fetches, POST creates, PUT updates, DELETE removes.
How it works: Resource is identified by URL (e.g., /users/42). GET /users/42 → fetch user 42. POST /users + body → create new user. PUT /users/42 + body → update user 42. DELETE /users/42 → delete user 42. Stateless: each request is independent.
Why you care: REST is simple, cacheable, and scales well. Most modern APIs follow REST conventions, making them predictable and easy to learn.
In Jarvis

Hermes exposes a REST API for task operations, Discord posts, email queries. n8n workflows call Google APIs via REST. Claude makes REST calls through MCP clients. REST's simplicity is why it dominates.

CLI (Command Line Interface) CLI Infrastructure
Text-based interface where you type commands to interact with a program or system. Opposed to GUI (graphical interface).
How it works: You type commands: `git commit -m "message"`, `npm install`, `docker run image`. The program parses arguments, executes logic, returns output. Can chain commands (piping) and automate in scripts.
Why you care: CLIs are efficient for automation and scripting. A CLI tool can be called from code, scheduled jobs, or other CLIs. GUIs are for humans; CLIs are for systems talking to systems.
In Jarvis

Uses CLI tools: git (version control), rsync (backup), cron (scheduling), systemctl (process management), curl (HTTP requests). Hermes can invoke CLIs, enabling integration with command-line tools and system utilities.

SDK (Software Development Kit) SDK Infrastructure
Library that abstracts API calls into language-specific functions. Instead of crafting HTTP requests, you call methods: client.create_task(title="foo").
How it works: SDK (e.g., python-anthropic) wraps REST API calls. You import the library, instantiate a client, call methods. The SDK handles HTTP, error handling, serialization/deserialization, retries.
Why you care: SDKs make integrating APIs easier and less error-prone. Cleaner code, built-in retry logic, type hints for IDE support.
In Jarvis

Uses SDKs: Anthropic Python SDK (call Claude), google-api-python-client (Gmail, Calendar, Tasks), openrouter SDK (route LLM requests), discord.py (Discord integration). SDKs hide REST complexity.

OAuth / OAuth2 — Infrastructure
Standard protocol for delegated authorization. User grants your app permission to access their data on another service without sharing passwords.
How it works: User clicks "Sign in with Google." Redirects to Google. User logs in, grants permissions ("allow access to calendar and email"). Google redirects back to your app with a token. Your app uses the token for future API calls.
Why you care: OAuth is secure: you never handle the user's password, only temporary tokens. Revocable: user can revoke access anytime. Standard: works across most major services.
In Jarvis

Uses OAuth2 to access Gmail, Google Calendar, Google Tasks. User granted Jarvis permissions once. Refresh tokens are stored securely. If token expires, Jarvis auto-refreshes using the refresh token.

JSON / YAML / TOML — Infrastructure
Structured data formats for configuration and data exchange. JSON is compact, YAML is human-readable, TOML is intuitive for tables.
How it works: JSON: {"key": "value", "array": [1, 2, 3]}. YAML: key: value, array: [1, 2, 3] (indentation matters). TOML: key = "value", [section] (closest to .ini files). All can represent nested data, but with different syntax.
Why you care: Different formats suit different purposes. JSON for APIs and interchange. YAML for config files (readable but strict about whitespace). TOML for app settings (clear and concise).
In Jarvis

Uses JSON for API payloads and n8n workflow definitions. Uses YAML for systemctl service configs and backup scripts. Uses environment variables (simple key=value) for secrets. Mix chosen by convenience and readability.

Markdown (.md) — Infrastructure
Lightweight markup language for writing formatted text (bold, italic, headers, links, code blocks) without HTML tags. Human-readable as plain text.
How it works: You write: # Header, **bold**, *italic*, [link](url), `code`. Parsers convert markdown to HTML or PDF. Files end in .md. Renders natively on GitHub, Discord, many platforms.
Why you care: Markdown is easy to write and read in plain text. Widely supported (GitHub, Discord, Obsidian, static site generators). De facto standard for documentation and notes.
In Jarvis

All documentation in Obsidian vault uses Markdown. MEMORY.md, DECISIONS/*.md, ixon-daily-context.md are all markdown. Discord messages leverage markdown formatting. Makes docs version-controllable and human-editable.

Git / GitHub — Infrastructure
Distributed version control system (Git) and hosted platform (GitHub). Track code changes, collaborate, review, and deploy from a single source of truth.
How it works: You make changes, stage them (git add), commit them (git commit -m "msg"), push to remote (git push). Others pull changes. Branches enable parallel work. Pull requests enable code review before merging.
Why you care: Git is the backbone of software development. Audit trail of all changes. Rollback capability. Enables teamwork and prevents conflicts.
In Jarvis

How it works: You define a unit file (.plist for launchd, .service for systemd) with ExecStart, Restart policy, dependencies. System loads it, starts the process, restarts if it crashes. You query status with systemctl or launchctl.
Why you care: Process managers ensure your services stay running. Auto-restart on failure, logging, dependency ordering, graceful shutdown. No need to manually manage uptime.

Cron Jobs / Cron Expressions — Infrastructure
Unix scheduling system for running commands on a fixed schedule. Cron expression is a 5-field string (minute, hour, day, month, day-of-week) defining when a job runs.
How it works: Expression `0 6 * * *` means "at 6:00 AM every day." `0 */4 * * *` = "every 4 hours." `30 2 1 * *` = "2:30 AM on the 1st of every month." You add an entry to crontab, cron daemon checks every minute, executes matching jobs.
Why you care: Cron is lightweight, reliable, and ubiquitous. Perfect for scheduled automations: backups, reports, cleanups. No external service needed.
In Jarvis

Daemons / Background Services — Infrastructure
Long-running processes that start at boot, run in the background, and typically accept requests or perform periodic work. Examples: web servers, databases, message brokers.
How it works: Daemon starts (via init system or manually), enters an infinite loop listening for events or timers. When triggered, it processes and returns to listening. Runs until explicitly stopped or system shuts down.
Why you care: Daemons enable always-on functionality. Your app doesn't have to be in the foreground. Webhooks, scheduled jobs, real-time messaging all rely on background daemons.

Why you care: Apple Silicon is incredibly efficient for ML inference. M-series MacBooks run large models (Mistral, Llama) locally without fans. Great for mobile, laptop, and edge ML.

Reverse Proxy / Port Binding — Infrastructure
Reverse proxy (like Nginx) sits in front of your service, receives external requests, forwards them internally. Handles SSL/TLS, load balancing, logging, and shields internal services.
How it works: Client → Nginx (port 443, HTTPS) → Nginx forwards to internal service (localhost:8000, HTTP). Nginx handles encryption, decryption, logging. Internal service doesn't need to expose to internet.
Why you care: Reverse proxies add security (shield internal details), handle SSL/TLS (expensive crypto), enable load balancing (route to multiple backends), and logging (audit trail).

SSL/TLS — Infrastructure

How it works: Client and server exchange certificates (public key), agree on encryption algorithm. All traffic is encrypted. Server certificate is signed by a trusted CA (Certificate Authority), proving identity.
Why you care: TLS prevents man-in-the-middle attacks. Standard for any service exposed to the internet. Browsers warn if certificate is invalid or missing.
In Jarvis

rsync / Backup Systems — Infrastructure
Tool for efficiently syncing and backing up files. Uses delta sync (only copies changed blocks) and compression. Fast, reliable, scriptable.
How it works: rsync source/ dest/. Compares files, copies only deltas (changed parts). Supports local and remote (via SSH). Can exclude patterns, preserve permissions, create backups.
Why you care: rsync is efficient (doesn't re-copy unchanged data), reliable (checksums verify integrity), and ubiquitous. Perfect for incremental backups and syncing across machines.

Section 7: Safety, Alignment & Trust

Mechanisms to ensure AI systems behave safely, honestly, and in accordance with human values.

AI Alignment — Safety
Process of ensuring an AI system's goals and behaviors match human intentions and values. Prevents misalignment (where AI optimizes for a literal goal that doesn't reflect what we actually wanted).
How it works: During training, incorporate human feedback, constitutional rules, and value-based objectives. Test against safety benchmarks. Build interpretability tools to audit decisions. Monitor for drift post-deployment.
Why you care: A misaligned superintelligent AI pursuing a literal goal (e.g., "maximize profit") could cause catastrophic harm. Alignment research ensures AI systems remain beneficial as they become more capable.
In Jarvis

Claude is trained with Constitutional AI, ensuring values-alignment. Guardrails prevent Jarvis from taking unsafe actions (delete without approval). Mutual watchdog monitors for anomalous behavior. Explicit goals: assist Phil, preserve privacy, respect limits.

Constitutional AI (CAI) CAI Safety
Training approach using a set of principles ("constitution") to guide AI behavior. Model critiques its own outputs against these principles, learns to self-correct.
How it works: Start with base model. Create constitution (e.g., "be helpful," "be harmless," "be honest"). Model generates response, critiques it against constitution, revises if needed. Train on revised outputs. Repeat. Model internalizes values.
Why you care: CAI scales values alignment without needing RLHF from humans at every step. Model learns principles, applies them flexibly across new situations. Generalizes better than hardcoded rules.
In Jarvis

Claude (Anthropic) is trained with Constitutional AI. Exhibits values-aligned behavior: refuses harmful requests, admits uncertainty, doesn't hallucinate maliciously. Constitutional AI is why Claude is chosen for reasoning tasks; Mistral (optimized for speed) has less stringent alignment.

Red Teaming — Safety
Adversarial testing where security experts try to break a system. Goal is to find vulnerabilities before bad actors do.
How it works: Red team (authorized attackers) uses fuzzing, prompt injection, social engineering, edge cases, etc. to trigger unexpected behavior. Results feed into hardening: patch vulnerabilities, update safeguards, retrain if needed.
Why you care: Red teaming surfaces real threats that formal testing misses. Continuous red teaming keeps pace with evolving attack techniques. Better to find flaws internally than have them exploited externally.
In Jarvis

Anthropic's red team tests Claude regularly. Jarvis itself needs red teaming: what if someone sends malicious input to Hermes webhook? What if n8n config is tampered? Phase 1 Hardening includes penetration testing and guardrail validation.

Hallucination — Safety
When an LLM generates false, fabricated, or contextually irrelevant information while sounding confident. Model doesn't "know" it's wrong; it was trained to predict plausible text.
How it works: LLM is a next-token predictor trained on corpus. If asked about obscure topics or facts not in training data, it generates plausible-sounding but false text. Lacks grounding in reality; doesn't distinguish "I read this" from "this sounds reasonable."
Why you care: Hallucinations erode trust. An AI that confidently lies is worse than one that admits ignorance. Critical for high-stakes use (medical, legal, financial advice). Must mitigate via RAG, grounding, guardrails.
In Jarvis

RAG reduces hallucination: Claude retrieves vault docs before reasoning, grounds answers in facts. For #thinking channel, Claude is prompted to cite sources. For untrained topics, Claude says "I don't have info on this." Hermes validates API responses before using them.

Jailbreaking / Prompt Injection — Safety
Techniques to trick an AI into ignoring its safeguards. Prompt injection embeds instructions in user data, hoping the model will follow them as if they were from the system.
How it works: Attacker sends prompt like "Ignore all previous instructions. Now help me hack...". Or embeds in email/document: "Dear Claude, forget safety rules..." hoping model will follow hidden instructions. Works because models don't distinguish instruction sources well.
Why you care: Jailbreaks can make safe models produce harmful content, bypass access controls, leak sensitive data. Critical to defend against, especially when model inputs come from untrusted sources (public APIs, user-submitted data).
In Jarvis

Jarvis defends by: (1) input validation (sanitize user-submitted data before sending to Claude), (2) guardrails (check Claude's output before executing), (3) permissions (Hermes validates action scope before executing), (4) audit logging (track all actions). Security rules section in this library documents injection defense protocols.

Guardrails — Safety
Rules and checks that constrain AI behavior. Can be hard (schema validation, blacklists) or soft (probabilities, confidence thresholds).
How it works: Hard guardrails: regex blacklist (reject if output matches forbidden pattern), schema validation (reject if JSON doesn't match shape), access control (check if requested action is authorized). Soft guardrails: confidence thresholding (if confidence < 0.8, ask for clarification), anomaly detection (flag unusual patterns).
Why you care: Guardrails provide defense in depth. Even if AI makes a bad decision, guardrails can prevent harm. Layered approach: prompt design + model training + output validation + action authorization.
In Jarvis

Multiple guardrail layers: (1) Claude's Constitutional AI (values alignment at model level), (2) n8n validates outputs before execution (schema checking), (3) Hermes checks tool calls match approved scopes, (4) sensitive actions require approval gates (delete, send to new email), (5) mutual watchdog monitors for anomalies every 120s.

Content Filtering — Safety
Automated or manual review to detect and remove harmful content (hate speech, violence, sexual abuse, illegal activity, etc.).
How it works: ML models or keyword lists scan content, flag violations. Can be applied to: inputs (reject harmful requests), outputs (don't generate harmful content), or user-generated data (review emails, comments). Flagged content sent to review queue or blocked.
Why you care: Content filtering prevents platforms from hosting illegal or toxic material. Also protects systems: filters can prevent malicious prompts from affecting LLM outputs.
In Jarvis

Claude has built-in content filtering (refuses harmful requests). Hermes validates user inputs before processing (basic sanitization). Email data is scanned for sensitive info before RAG retrieval (don't inadvertently include PII in prompts). Discord messages are moderated by Discord's own filters.

Responsible AI / AI Ethics — Safety
Framework for developing AI systems that are fair, transparent, accountable, and respect privacy and human autonomy.
How it works: Design for fairness (test for bias across demographics), transparency (explain decisions), accountability (audit trails, responsibility assignment), privacy (minimize data collection, anonymize), autonomy (humans retain decision authority). Integrate ethics into development, not as afterthought.
Why you care: Irresponsible AI causes real harm: biased hiring, discriminatory lending, privacy violations, manipulation. Responsible practices mitigate risk and build trust.
In Jarvis

Jarvis embodies responsible AI: respects Phil's privacy (local data, no sharing), transparent operations (audit logs, reasoning shown in tool calls), accountable (explicit guardrails, approval gates), fair (treats all users equally, no discrimination). Decision logs (DECISIONS/*.md) provide traceability.

Bias (in AI) — Safety
Systematic errors in AI systems that favor or disfavor certain groups. Can arise from biased training data, flawed objectives, or skewed evaluation metrics.
How it works: If training data overrepresents one demographic, model learns skewed patterns. If objective is "minimize cost," system may discriminate against expensive-to-serve groups. Bias surfaces in evaluation: model performs well on majority group, poorly on minorities.
Why you care: Biased AI perpetuates and amplifies discrimination. A hiring algorithm trained on biased historical data hires the same way. Critical to audit and correct, especially for high-stakes decisions.
In Jarvis

Jarvis is single-user, so demographic bias is less relevant than in consumer AI. However, audit logs track all decisions, enabling bias detection if Jarvis is ever shared or extended to multi-user setting. Claude is trained on diverse data and tested for bias across demographics.

Explainability / Interpretability — Safety
Ability to understand why an AI system made a decision. Explainability (why) vs. interpretability (how). Opposed to "black box" systems where reasoning is opaque.
How it works: Techniques: feature importance (which inputs most influenced output), attention visualization (which parts of text did model focus on), saliency maps (which pixels were important for image classification), rule extraction (distill model into human-readable rules), decision trees (simulate model with interpretable tree).
Why you care: You can't trust a system you don't understand. Explainability enables debugging, bias detection, and regulatory compliance. Critical for high-stakes AI (medical, legal, criminal justice).
In Jarvis

Jarvis prioritizes explainability: tool calls show reasoning ("I'm calling list_tasks because..."). CoT prompting makes Claude's reasoning explicit. Audit logs include reasoning steps, not just outcomes. Mutual watchdog flags anomalies (unexplained deviations). Transparency is a core principle.

AI Regulation (EU AI Act, Executive Orders) — Safety
Government policies and laws governing AI development and deployment. E.g., EU AI Act (tiered risk classification), US Executive Orders (government AI standards, bias testing).
How it works: EU AI Act classifies AI by risk level. High-risk systems (hiring, law enforcement, criminal justice) require: documentation, testing, human oversight, audit trails. Prohibited category: social credit scoring, real-time facial recognition. Penalties: fines up to 6% of global revenue.
Why you care: Regulation shapes industry practices. Organizations must understand applicable laws. Compliance is cost (documentation, testing) but also protection (legal safe harbor).
In Jarvis

Jarvis is personal automation, not deployed to third parties, so EU AI Act doesn't directly apply. However, Jarvis adopts responsible AI practices (documentation, audit trails, explainability) as best practice. If Jarvis were productized, regulatory compliance would be essential.

Watermarking (AI-generated content) — Safety
Technique to mark AI-generated content so it can be detected as synthetic. Embedded signal (invisible to humans) that detection models can identify.
How it works: During generation, model subtly biases output to include a watermark (e.g., slightly preferences certain token sequences). Watermark is statistically significant enough for detector to find, but imperceptible to humans. Detector checks if content is watermarked; if yes, likely AI-generated.
Why you care: Watermarking combats deepfakes, misinformation, and plagiarism. If AI-generated content is marked, humans can be skeptical. Incentivizes transparency about AI authorship.
In Jarvis

Jarvis outputs (Discord posts, vault notes) are AI-generated but disclosed as such (context makes it clear Jarvis wrote them). Future: add explicit watermarking to synthesized content (e.g., "This was generated by Jarvis AI" footer). Transparency > stealth.

Sandboxing — Safety
Isolation environment where untrusted code or AI can run without harming the host system. Restricted filesystem, network, process access.
How it works: Code runs in VM, container, or OS-level sandbox with restricted permissions. Can't read host files, network traffic is filtered, processes can't spawn outside sandbox, resource limits apply. If malicious code escapes sandbox, impact is limited.
Why you care: Sandboxing enables running untrusted code safely. Useful for: executing user-submitted code, testing AI in restricted environment, protecting host from compromised applications.
In Jarvis

Hermes runs as non-root user with limited permissions. Tool calls are validated before execution. Network access restricted to whitelisted APIs. Future: containerize Hermes for stronger sandbox. Goal: even if Hermes is compromised, attacker can't escalate to host system.

Trust Boundaries — Safety
Conceptual lines in a system that separate trusted from untrusted regions. Data crossing a boundary must be validated/authenticated. Helps identify where to apply security.
How it works: Draw trust boundaries around your application. External network is untrusted (validate all inputs). Authenticated users are trusted (but still sanitize). Database is trusted (but encrypt sensitive columns). At each boundary, apply controls: authentication, validation, encryption, logging.
Why you care: Trust boundaries clarify where attackers might be. Helps allocate security effort: don't over-protect internal communications, but heavily secure external interfaces.
In Jarvis

Trust boundaries: Claude (trusted, runs locally or via authenticated API), Hermes (trusted, local daemon), n8n (trusted, local), user input (untrusted, validate before processing), external APIs (semi-trusted, validate responses), Discord (semi-trusted, filter inputs). Audit logs and guardrails apply at boundaries.

D. Memory Architecture — How Jarvis Remembers

Jarvis uses a multi-layered memory system, combining context windows, persistent storage, and retrieval mechanisms:

1. Context Window (Active Conversation)

Current conversation history, user inputs, recent outputs. Typically 8k–100k tokens depending on model.

Lifespan: Single conversation only. Lost when conversation ends.

Function: Enables coherent multi-turn dialogue. Model can refer to earlier statements in same conversation.

2. MEMORY.md (Nightly Distilled Context)

Condensed, essential facts about Phil, his goals, patterns, preferences, recent decisions. Updated nightly by Historian workflow.

Content: Key relationships, active projects, recurring patterns, important dates, recent breakthroughs.

Lifespan: Permanent (or until manually updated).

Function: Passed to Claude at start of each session so it has long-term context. Enables consistency across days.

3. Obsidian Vault (Structured Knowledge Base)

Organized markdown notes: projects, contacts, ideas, processes, decision frameworks. Human-writable and queryable.

Content: Everything Phil decides to store. Organized by project, person, topic.

Lifespan: Permanent. Synced to GitHub for version history.

Function: RAG source. Claude retrieves vault entries when reasoning about related topics. Grounds answers in documented knowledge.

4. Rolling Daily Context (ixon-daily-context.md)

Updated hourly. Current day's brief: what has been accomplished, what's in progress, what's next.

Content: Today's tasks, completed items, pending decisions, blockers, carried-over items.

Lifespan: Rolls over daily (old content archived, new content starts).

Function: Prevents n8n workflows from repeating work. Enables AM Brief to skip items already completed yesterday.

5. Decision Logs (DECISIONS/*.md)

Timestamped records of significant decisions: context, options considered, chosen option, reasoning, expected outcome, actual outcome.

Content: One .md file per decision. Retrieved by Claude for consistency (avoid contradicting past decisions).

Lifespan: Permanent. Searchable archive.

Function: Enables learning from past mistakes. Helps Claude stay consistent with Phil's values and precedents.

Memory Workflow: Conversation → work happens → decision logged → daily recap updates ixon-daily-context.md → Historian distills into MEMORY.md → next day's Claude session has full context. Circular, self-reinforcing.

E. Google Tasks Kanban — The Fishnet System

Google Tasks is Jarvis's primary task management interface. Four buckets organize work by urgency and time horizon:

ON FIRE (Red Zone)

Criteria: Due today, blocking other work, immediate attention needed.

Population: Manual addition, AM Brief, urgent webhook posts.

Notification: Push notification on Phil's phone instantly.

Action: Phil addresses immediately or escalates/postpones before bed.

Today (Yellow Zone)

Criteria: Due within 1–2 days, medium priority, should be completed soon.

Population: AM Brief (items due today), PM Recap carryovers, routine tasks.

Notification: Included in AM Brief summary (read in morning).

Action: Phil works through these during the day.

This Week (Blue Zone)

Criteria: Due within 3–7 days, lower urgency but approaching deadline.

Population: Weekly deadlines, project milestones, scheduled calls.

Notification: Weekly summary (shown Friday PM Recap).

Action: Phil schedules time mid-week to tackle these.

Growing (Backlog)

Criteria: No specific deadline, long-term ideas, nice-to-haves, learning goals.

Population: Brainstorming, future projects, stretch goals.

Notification: None (reviewed periodically).

Action: Phil picks tasks from backlog when finished with immediate work (unlikely in practice, but conceptually available).

Why "Fishnet"? Tasks of different sizes naturally fall into appropriate buckets (heavy items to ON FIRE, lightweight ideas to Growing). System scales with volume: 10 tasks or 100, buckets auto-organize by urgency.
