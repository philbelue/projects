01
Gap One
What Runtime IS Hermes?
“Opus designed the interior of a house without picking what it’s built from.”

Fair. The blueprint described skills, handoff protocols, and self-repair cascades without ever answering the most fundamental question: what process runs, what language, what framework, and why. Here’s the answer.

The Decision

Hermes is a Python 3.12+ asyncio daemon, managed by launchd, running as a user-level LaunchAgent. Single process. No framework. No container. Just Python, asyncio, and a handful of well-tested libraries.

Why Python over every alternative:

OptionProsConsVerdict
Python asyncio daemon Native async/await. Phil’s AI stack is Python. psutil, aiosqlite, aiohttp all mature. Single process, single plist. Ollama client libraries are Python-native. GIL limits CPU-bound work (irrelevant—Hermes is I/O-bound). CHOSEN
Node.js process n8n is Node. Event loop is good. Phil’s AI tooling is Python. Two ecosystems to maintain. Node process monitoring libraries less mature on macOS. REJECTED
Shell script loop Simple. No dependencies. No async. No error handling. No state management. No task queue. Fragile. REJECTED
OpenClaw skill Already running infrastructure. Hermes monitors OpenClaw. If Hermes IS OpenClaw, it can’t monitor itself. Circular dependency. Single point of failure. REJECTED

The main.py skeleton — this is the actual process that runs:

Python ~/jarvis/hermes/main.py 
#!/usr/bin/env python3
"""Hermes — Jarvis infrastructure watchdog daemon."""

import asyncio
import signal
import logging
import sys
from pathlib import Path

from core.queue import TaskQueue
from core.monitor import HealthMonitor
from core.state import DaemonState
from core.discord import DiscordAlert
from skills.health_probe import HealthProbeSkill
from skills.config_guardian import ConfigGuardianSkill
from skills.workflow_exec import WorkflowExecutorSkill

HERMES_HOME = Path.home() / "jarvis" / "hermes"
HEALTH_PORT = 7777
BOOT_GRACE  = 120  # seconds before monitoring goes live

logger = logging.getLogger("hermes")

class Hermes:
    def __init__(self):
        self.state    = DaemonState()
        self.queue    = TaskQueue(HERMES_HOME / "data" / "queue.db")
        self.discord  = DiscordAlert()
        self.skills   = {}
        self._shutdown = asyncio.Event()

    async def start(self):
        """Boot sequence: init → grace period → operational."""
        logger.info("Hermes starting — entering boot grace period (%ds)", BOOT_GRACE)
        self.state.set("booting")

        # Initialize subsystems
        await self.queue.init_db()
        self.skills = {
            "health_probe":    HealthProbeSkill(),
            "config_guardian": ConfigGuardianSkill(),
            "workflow_exec":   WorkflowExecutorSkill(),
        }

        # Register signal handlers (SIGTERM from launchd)
        loop = asyncio.get_event_loop()
        for sig in (signal.SIGTERM, signal.SIGINT):
            loop.add_signal_handler(sig, lambda s=sig: self._handle_signal(s))

        # Boot grace: poll dependencies but don't alert on failures
        await self._boot_grace_period()

        # Go operational
        self.state.set("operational")
        logger.info("Hermes operational — all systems monitored")
        await self.discord.send("Hermes is online and monitoring.")

        # Launch concurrent loops
        await asyncio.gather(
            self._health_loop(interval=300),     # 5 min health checks
            self._queue_processor(),               # task queue consumer
            self._health_endpoint(HEALTH_PORT),    # HTTP /health for launchd
            self._shutdown.wait(),                 # blocks until signal
        )

    async def _boot_grace_period(self):
        """Wait up to BOOT_GRACE seconds for dependencies.
        Poll every 5s. Don't alert on failures during this window."""
        deadline = asyncio.get_event_loop().time() + BOOT_GRACE
        while asyncio.get_event_loop().time() < deadline:
            deps = await self._check_dependencies()
            if deps["all_ready"]:
                logger.info("All dependencies ready — grace period complete")
                return
            await asyncio.sleep(5)
        logger.warning("Grace period expired — starting with available deps")

    async def _health_loop(self, interval: int):
        """Run health probe skill on schedule."""
        while not self._shutdown.is_set():
            result = await self.skills["health_probe"].execute()
            if result.has_failures:
                await self.discord.send(result.format_alert())
                await self._attempt_self_repair(result.failures)
            await asyncio.sleep(interval)

    async def _queue_processor(self):
        """Claim and execute tasks from SQLite queue."""
        while not self._shutdown.is_set():
            task = await self.queue.claim()
            if task is None:
                await asyncio.sleep(5)  # poll interval
                continue
            try:
                skill = self.skills[task.skill_name]
                await skill.execute(task.payload)
                await self.queue.complete(task.id)
            except Exception as e:
                await self.queue.fail(task.id, str(e))

    async def _health_endpoint(self, port: int):
        """HTTP /health for launchd and external probes."""
        async def handler(reader, writer):
            status = self.state.current
            code = b"200" if status == "operational" else b"503"
            body = f'{{"status":"{status}"}}'.encode()
            writer.write(
                b"HTTP/1.1 " + code + b" OK\r\n"
                b"Content-Type: application/json\r\n"
                b"Content-Length: " + str(len(body)).encode() + b"\r\n\r\n"
                + body
            )
            await writer.drain()
            writer.close()
        server = await asyncio.start_server(handler, "127.0.0.1", port)
        async with server:
            await server.serve_forever()

    def _handle_signal(self, sig):
        logger.info("Signal %s received — graceful shutdown", sig)
        self._shutdown.set()

if __name__ == "__main__":
    logging.basicConfig(
        level=logging.INFO,
        format="%(asctime)s [%(name)s] %(levelname)s: %(message)s",
        stream=sys.stdout,
    )
    asyncio.run(Hermes().start())

02
Gap Two
The Task Queue: SQLite Replaces fswatch
“fswatch is not a task queue — it’s a race condition waiting to happen.”

Jarvis is right. Watching a directory of markdown files with fswatch gives you zero atomicity, zero dead letter handling, and zero idempotency. Two handoffs land at the same time? Race. Hermes crashes mid-execution? File stuck at in_progress forever. NVMe not mounted? Handoff directory doesn’t exist. Silent failure.

The Decision

SQLite in WAL mode is the task queue. Single file, atomic transactions, no external infrastructure. The Obsidian vault gets markdown mirrors for human readability, but the source of truth is queue.db. No fswatch anywhere in the system.

Python ~/jarvis/hermes/core/queue.py 
"""SQLite WAL task queue with atomic claiming and dead letter handling."""

import asyncio
import json
import aiosqlite
from dataclasses import dataclass
from typing import Optional
from pathlib import Path

@dataclass
class Task:
    id: int
    skill_name: str
    payload: dict
    attempts: int

class TaskQueue:
    def __init__(self, db_path: Path):
        self.db_path = str(db_path)
        self._lock = asyncio.Lock()  # serialize writes

    async def init_db(self):
        async with aiosqlite.connect(self.db_path) as db:
            await db.execute("PRAGMA journal_mode=WAL")
            await db.execute("PRAGMA synchronous=NORMAL")
            await db.execute("PRAGMA busy_timeout=30000")
            await db.execute("""
                CREATE TABLE IF NOT EXISTS tasks (
                    id          INTEGER PRIMARY KEY AUTOINCREMENT,
                    skill_name  TEXT NOT NULL,
                    payload     TEXT NOT NULL DEFAULT '{}',
                    status      TEXT NOT NULL DEFAULT 'pending',
                    idempotency TEXT UNIQUE,
                    attempts    INTEGER DEFAULT 0,
                    max_retries INTEGER DEFAULT 3,
                    created_at  TEXT DEFAULT (datetime('now')),
                    claimed_at  TEXT,
                    finished_at TEXT,
                    error       TEXT
                )
            """)
            await db.execute("""
                CREATE INDEX IF NOT EXISTS idx_status
                ON tasks(status, created_at)
            """)
            await db.commit()

    async def enqueue(self, skill: str, payload: dict,
                      idempotency: str = None) -> int:
        """Add a task. Idempotency key prevents duplicates."""
        async with self._lock:
            async with aiosqlite.connect(self.db_path) as db:
                try:
                    cur = await db.execute(
                        "INSERT INTO tasks (skill_name, payload, idempotency) VALUES (?, ?, ?)",
                        (skill, json.dumps(payload), idempotency)
                    )
                    await db.commit()
                    return cur.lastrowid
                except aiosqlite.IntegrityError:
                    return -1  # duplicate idempotency key — skip

    async def claim(self) -> Optional[Task]:
        """Atomically claim the oldest pending task.
        Uses BEGIN IMMEDIATE to prevent read-upgrade deadlocks."""
        async with self._lock:
            async with aiosqlite.connect(self.db_path) as db:
                await db.execute("BEGIN IMMEDIATE")
                try:
                    cur = await db.execute("""
                        SELECT id, skill_name, payload, attempts
                        FROM tasks
                        WHERE status = 'pending'
                        ORDER BY created_at ASC
                        LIMIT 1
                    """)
                    row = await cur.fetchone()
                    if not row:
                        await db.execute("ROLLBACK")
                        return None

                    task_id, skill, payload, attempts = row
                    await db.execute("""
                        UPDATE tasks
                        SET status = 'in_progress',
                            attempts = attempts + 1,
                            claimed_at = datetime('now')
                        WHERE id = ? AND status = 'pending'
                    """, (task_id,))
                    await db.commit()

                    return Task(
                        id=task_id,
                        skill_name=skill,
                        payload=json.loads(payload),
                        attempts=attempts + 1
                    )
                except:
                    await db.execute("ROLLBACK")
                    raise

    async def complete(self, task_id: int):
        async with self._lock:
            async with aiosqlite.connect(self.db_path) as db:
                await db.execute("""
                    UPDATE tasks SET status='completed', finished_at=datetime('now')
                    WHERE id=?
                """, (task_id,))
                await db.commit()

    async def fail(self, task_id: int, error: str):
        """Fail a task. If max retries exceeded → dead_letter."""
        async with self._lock:
            async with aiosqlite.connect(self.db_path) as db:
                cur = await db.execute(
                    "SELECT attempts, max_retries FROM tasks WHERE id=?",
                    (task_id,)
                )
                row = await cur.fetchone()
                attempts, max_retries = row

                if attempts >= max_retries:
                    new_status = "dead_letter"
                else:
                    new_status = "pending"  # back in queue for retry

                await db.execute("""
                    UPDATE tasks
                    SET status=?, error=?, finished_at=datetime('now')
                    WHERE id=?
                """, (new_status, error, task_id))
                await db.commit()

    async def reap_stale(self, stale_minutes: int = 10):
        """Find tasks stuck in 'in_progress' for too long.
        Reset to 'pending' for retry or 'dead_letter' if exhausted."""
        async with self._lock:
            async with aiosqlite.connect(self.db_path) as db:
                await db.execute(f"""
                    UPDATE tasks
                    SET status = CASE
                        WHEN attempts >= max_retries THEN 'dead_letter'
                        ELSE 'pending'
                    END,
                    error = 'stale: exceeded {stale_minutes}m in_progress'
                    WHERE status = 'in_progress'
                    AND claimed_at < datetime('now', '-{stale_minutes} minutes')
                """)
                await db.commit()

Gap Three
Self-Improvement: The Honest Answer
“That’s marketing, not engineering.”

Jarvis is right again. The blueprint said every skill “self-improves” — recalibrates thresholds, builds dangerous-keys lists, tracks success rates. That sounds compelling and is completely unspecified. So here’s the honest answer:

The Decision

Hermes does NOT self-improve in Phases 1–2. Period. Self-modification of a system that monitors your infrastructure is a Phase 3+ capability, and it requires a human-in-the-loop gate. Anything else is asking for a bad threshold change to cascade into a false-positive storm that wakes you up at 3am.

Phase 1–2: Static skills with manual tuning. All thresholds live in config.yaml. You change them by editing the file. Hermes watches config.yaml with Config Guardian and applies changes on next cycle. That’s it. No AI analyzing its own behavior. No auto-recalibration.

Phase 3+ (May onwards): The actual self-improvement mechanism, if you choose to build it:

Architecture Self-improvement loop (Phase 3 only)

          AVG(duration_ms) as avg_duration,
          GROUP_CONCAT(DISTINCT error) as errors
   FROM task_log
   WHERE finished_at > datetime('now', '-7 days')
   GROUP BY skill_name

2. ANALYZE
   # n8n workflow calls Ollama/Mistral (local, no cloud dependency)
   # Input: weekly execution summary (above SQL output)
   # Output: JSON with proposed config changes
   # Example: {"health_probe.interval": 180, "reason": "12 false negatives this week"}

3. PROPOSE (not apply)

    Reason: 12 false negatives this week at 300s interval.
    React ✅ to approve, ❌ to reject."

4. GATE
   # Human approves via Discord reaction
   # Bot reads reaction, writes approved changes to config.yaml
   # Config Guardian detects change, applies on next cycle

5. ROLLBACK
   # config.yaml is git-versioned (~/jarvis/hermes/ is a git repo)
   # If next week's metrics are worse:
   git log --oneline config.yaml  # find last known good
   git checkout abc123 -- config.yaml  # revert
   # Hermes proposes the rollback the same way — via Discord approval

What prevents a bad self-modification from cascading?

Three things: (1) The human gate — nothing auto-applies. (2) Git versioning — every config change has a revert target. (3) Weekly cadence — you have 7 days of data before the next proposal. No real-time feedback loop. No oscillation possible.

Gap Four
The Startup Deadlock, Solved
“This is a classic distributed systems startup problem and the doc skips it entirely.”

Hermes monitors Jarvis (OpenClaw). Jarvis monitors Hermes. Both managed by launchd. On cold boot, launchd starts both simultaneously. The naive implementation: both start, neither is ready, both mark the other as down, restart cascade begins. Classic deadlock.

The Solution: Boot Grace Period + State Machine

Adapted from Kubernetes startup/readiness/liveness probe separation. Hermes has four states, and it does not monitor anything until it has passed through the boot grace period.

The state machine:

BOOTING — grace period (120s) polls deps, no alerts → OPERATIONAL
OPERATIONAL — 3 consecutive health failures on a dep → DEGRADED
DEGRADED — dep recovers (1 successful check) → OPERATIONAL
DEGRADED — 5 min with no recovery → EMERGENCY — Discord alert + stop trying →

The cold boot sequence, concretely:

T+0s launchd loads all plists. OpenClaw, Ollama, n8n, and Hermes all start. No guaranteed ordering.
T+0s Hermes enters BOOTING state. Initializes SQLite queue, loads config. Does NOT check any service health yet.
T+5s Grace period polling begins. Every 5 seconds, Hermes tries to reach OpenClaw (:3000/health), n8n (:5678/healthz), Ollama (:11434/api/tags). Failures are logged but not acted on.
T+15s Ollama responds. n8n responds. OpenClaw still loading extensions.
T+35s OpenClaw responds. All deps ready. Grace period ends early.
T+35s Hermes enters OPERATIONAL state. Monitoring loops go live. Discord message: "Hermes online."
T+120s (Fallback) If any dep still isn’t up by 120s, Hermes goes OPERATIONAL anyway with those services marked DEGRADED. Monitoring is live. Alerts fire for whatever’s still down.

The specific deadlock scenario Jarvis raised, solved:

Scenario: Hermes starts before OpenClaw is ready

Old behavior (broken): Hermes starts, checks OpenClaw, fails, marks it down, tries to restart it, but OpenClaw is still booting. Restart collides with the existing boot. Chaos.

New behavior (fixed): Hermes starts, enters BOOTING state for 120s. During BOOTING, it polls OpenClaw every 5s but takes no action on failures. OpenClaw finishes booting at T+35s. Hermes detects it, transitions to OPERATIONAL. No false restart. No cascade.

Edge case: OpenClaw genuinely dead (not just slow to boot)? After 120s grace, Hermes goes OPERATIONAL with OpenClaw in DEGRADED. Three more failures → EMERGENCY. Discord alert. Hermes does not try to restart OpenClaw — launchd owns restarts. Hermes only reports.

Critical Design Rule

Hermes never restarts other services. That’s launchd’s job. Hermes monitors and alerts. If it tried to restart services, it would fight with launchd’s own KeepAlive restart logic, creating the exact oscillation Jarvis warned about. Separation of concerns: launchd restarts, Hermes watches and reports.

This eliminates the “mutual self-repair” startup deadlock entirely. There is no mutual restart. There’s one restart authority (launchd) and one monitoring authority (Hermes). They don’t compete.

What about Jarvis monitoring Hermes?

Jarvis (OpenClaw) checks http://127.0.0.1:7777/health on a 5-minute cron. If Hermes is down, Jarvis posts to Discord: “Hermes health check failed.” That’s it. No restart attempt. launchd handles the restart. The “mutual monitoring” is really just mutual awareness with unilateral action by launchd.

Errata
Corrections to Strategic Blueprint v1.0

Based on Jarvis review and Phil’s threat model assessment, these items are updated from the original blueprint.

DROP
NVMe Encryption

Threat model doesn’t justify it. Phil’s physical security is solid (key fob building, key fob apartment). FileVault on the internal SSD covers the real risk. The data on the NVMe is models and vault content, not financial credentials. The half-day reformat + re-download is not worth the marginal security gain.

DROP
SQLite → PostgreSQL Migration

Jarvis was right: this is resume-driven architecture. You have 5 workflows. SQLite is fine until you need queue mode or concurrent writers, and with a single Hermes daemon doing all the writing, you never will. Removing from roadmap entirely.

MOVE
Self-Improvement Skills → Phase 3+

Moved from Phase 2 to Phase 3 minimum. Requires human-in-the-loop approval gate, git-versioned configs, and weekly cadence. See Gap 3 above for full spec.

ADD
Eero Plus Subscription

$10/month for network-level threat detection on a system running AI infrastructure and deal intelligence. Cheap insurance. Adds intrusion detection, ad blocking at the DNS level, and threat scanning for all devices. Do it.

ADD
OpenClaw allowInsecureAuth Flag

Blueprint mentioned flipping this but didn’t flag the risk. Changing this flag may break Discord delivery if OpenClaw’s auth negotiation depends on it. Test in isolation first — spin up OpenClaw on a different port, flip the flag, verify Discord webhook delivery still works. Only then apply to production. Adding as explicit Phase 1 test item.

ADD
Hermes Does Not Restart Services

Blueprint implied Hermes would restart failed services (cross-agent repair). This is wrong. launchd owns all restarts via KeepAlive. Hermes monitors and alerts only. Fighting with launchd’s restart logic creates oscillation. Single authority for restarts. See Gap 4.

KEEP
psutil CPU Monitoring Workaround

Note for Hermes development: psutil.cpu_percent() is broken on Apple Silicon (reports 0–10% under load). Hermes health probe must use subprocess.run(["ps", "-p", str(pid), "-o", "%cpu="]) as fallback for per-process CPU. System-wide cpu_times_percent() works fine.
