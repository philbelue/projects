Executive Summary

The vault is not a notes folder. It's the shared brain for the entire Jarvis ecosystem. Today's failures trace to one root cause: the vault isn't being used as a memory system.

The Problem
Every time Phil switches channels, opens a new session, or comes back from a walk — both Jarvis and Hermes are amnesiacs. Phil carries the context in his head because the system doesn't. That's backwards.

What Happened Today

  • 3-hour wasted morning debugging non-issues (Discord permissions false positive)
  • Context bleed across channels — conversation in #local-chat invisible to #jarvis-builds
  • Repeated mistakes logged once, never caught again
  • $16 unauthorized spend on Claude Code API (OpenClaw CLI without approval)

The Fix

This spec makes the vault the single source of truth that both agents read from and write to on every session start and end.

Core Principle
The vault is not a convenience. It's a dependency. Neither Jarvis nor Hermes can function effectively without it.

SCHEMA.md — Vault Constitution

This is the single file that governs all vault operations. Both Jarvis and Hermes must read it before writing anything.

File Naming Rules

  • Lowercase with hyphens: ixon-pipeline-status.md
  • Dates in filenames: 2026-04-13.md
  • Decision files: D-001-gemini-flash-for-synthesis.md
  • No spaces, no special characters

Session Bridge Protocol

The #1 fix for context bleed. This is what makes Phil stop being the one carrying context in his head.

How It Works

File: OPS/working-context.md

On every conversation end (session close, channel switch, user goes AFK), both Jarvis and Hermes append a handoff block:

---
## [2026-04-13 09:32] Jarvis → #heremes-ops
**Status:** Fires resolved, Hermes permissions fixed
**What happened:**
- Fixed 3 morning fires (model name, brief routing, cross-channel memory)
- Hermes Discord permissions updated (View Channels, Add Reactions)
- Phil stressed, took a walk with Simba
**Decisions made:**
- All n8n synthesis must use google/gemini-2.0-flash via OpenRouter
- Never use Opus via API
**Open items:**
- OpenClaw upgrade to v2026.4.12 (not today, wait for stability)
- SCHEMA.md needs to be written
**Phil's mood:** Frustrated. Be efficient. Don't waste his time.
---

On Every Session Start

  1. Read OPS/working-context.md — last 3 entries
  2. Read OPS/mistakes.md — full file
  3. Read OPS/open-loops.md — all active loops
  4. Greet Phil with awareness: "I see you were working on X in #channel. Picking up from there?"
Rules
• Max 5 lines per section
• Include Phil's emotional state (this matters — don't poke a frustrated Phil)
• Include channel context
• Timestamp everything

Mistakes Protocol

The compounding error prevention system. Every mistake costs Phil once, never twice.

File: OPS/mistakes.md

Example Format

# Mistakes Log — Append Only

## M-012 | 2026-04-13 | Jarvis | SEVERITY: HIGH
**What happened:** Didn't check DM thread before responding.
**Root cause:** Not reading all available context before responding.
**Cost to Phil:** 15 min of frustration, wasted tokens
**Rule:** ALWAYS check DMs, threads, recent messages in ALL channels.

## M-011 | 2026-04-13 | Jarvis | SEVERITY: CRITICAL
**What happened:** Reported incorrect Discord permissions table.
**Root cause:** Decoded permission bits without accounting for inheritance.
**Cost to Phil:** Trust erosion, 10 min debugging
**Rule:** TEST permissions (try the action) not just read bit flags.

Protocol

  • Read at session start before doing anything
  • New mistakes appended immediately, not batched
  • Severity levels: LOW (annoyance), MEDIUM (wasted time), HIGH (wasted money/trust), CRITICAL (both)
  • Each entry includes a concrete RULE that prevents recurrence
  • File is append-only — never edited or trimmed
Critical
This file is not optional. It's the only system that prevents repeated failures. If you make a mistake that's already logged, you catch yourself before Phil has to.

Brain Cron — Nightly Maintenance

Automated vault health. Runs every night at midnight.

Tasks

  1. Session flush: Scan day's conversations, file decision-worthy items into WIKI/
  2. Orphan check: Find wiki pages with zero inbound links, flag them
  3. Stale loop check: Flag open loops older than 7 days with no updates
  4. Mistakes review: Count by severity for the week, add summary to daily log
  5. Index update: Rebuild HOME.md dashboard with current counts and recent activity

Implementation

  • n8n workflow, scheduled midnight
  • Uses Gemini Flash (free via OpenRouter)
  • Writes results to LOG/[date].md daily log
  • Flags urgent items to #jarvis-builds Discord channel
Automation
This is how Phil wakes up to a vault that's been organized, checked, and ready. Zero manual work.

Success Criteria

How Phil knows this is working:

  • Phil opens Discord after a walk with Simba → Jarvis greets him with context from the last conversation, not "what are we working on?"
  • Phil mentions something in #local-chat → switches to #jarvis-builds → Jarvis knows what was discussed
  • Jarvis makes a mistake that's already logged → Jarvis catches itself before Phil has to
  • A project Phil touched for 5 minutes last week → he asks about it → Jarvis finds the open loop and resumes
  • Phil asks "what did we spend this week?" → Jarvis has the answer from agent-log, not reconstructing chat history
  • Zero "I don't have context on that" responses from either bot
The Vision
Phil is never the one carrying context in his head again. The vault is the brain. Both agents read from it, write to it, and execute against it. The system remembers what Phil does, learns from what Phil mistakes, and never wastes Phil's time asking for context it already has.
