Executive Brief

You have a Mac Studio M4 Max running OpenClaw as the nervous system, n8n as the automation layer, Ollama for local inference, Obsidian as the knowledge vault, and Discord/Telegram as the communication channels. The foundation works. But it was broken an hour ago, and the first real reliability test hasn’t fired yet.

This blueprint addresses five things simultaneously: activating Hermes as a parallel executor with a concrete skill library and handoff protocol, hardening every service for production-grade reliability, implementing zero-trust security across the stack, building disaster recovery that lets you rebuild the entire system in under an hour, and wiring monitoring that tells you something’s wrong before you notice it yourself.

The Five Pillars

Interoperability — Every component talks to every other component through defined protocols, not ad-hoc wiring. Jarvis, Hermes, n8n, Obsidian, and Claude all share a common handoff schema and vault structure.

Stability — Services survive reboots, power loss, and crashes without human intervention. launchd plists configured correctly. Dependency chains verified. Cold boot tested.

Reliability — Workflows fire when they’re supposed to. Failures are caught, logged, and escalated. No more silent failures. No more “it was broken and nobody knew.”

Security — Zero-trust posture. Services bind to localhost. Encrypted at rest and in transit. API keys in Keychain, not config files. Reverse proxy with auth in front of everything.

Agility — The system evolves without breaking. Git-versioned configs. Rollback in seconds. New skills deploy without touching the core. Blue-green model upgrades.

Design Principle

Every layer is independently restartable. If Hermes dies, Jarvis keeps routing. If n8n dies, Jarvis still responds on Discord. If Ollama dies, inference falls back to OpenRouter then Anthropic. If the NVMe disconnects, internal SSD configs keep the core running. No single point of failure can take the whole organism down.

Hermes — Executor Architecture

Hermes is not another chatbot. It’s a task execution engine that sits underneath Jarvis. Jarvis decides what needs to happen. Hermes figures out how to do it, does it, and learns from the result. The relationship is command-and-control: Jarvis issues directives, Hermes executes them, reports back, and stores the pattern for next time.

Phase 1 — Foundation Base stack stable. AM/PM briefs firing. Cold boot tested. You are here
Phase 2 — Hermes Deploy launchd plist. First 3 skills. Handoff protocol live. Mutual monitoring.
Phase 3 — Evolution Self-improving skills. Parallel execution. Obsidian Publish dashboard.

First 3 Skills — What Hermes Learns First

Skills are not scripts. They’re reusable execution patterns that Hermes abstracts from successful task completions. Each skill has a trigger condition, execution steps, success criteria, and a self-improvement hook. Start with three skills that cover the highest-leverage operational patterns.

Skill 01: System Health Probe

Infrastructure

Comprehensive health check across all services with escalation logic.

  • Trigger: Cron (every 5 min) or on-demand via Jarvis command
  • Checks: Ollama (/api/tags + latency probe), n8n (/api/v1), OpenClaw (port 18789), Obsidian vault mount, NVMe availability, memory pressure, disk space
  • Escalation: If any check fails twice in a row → attempt automatic restart via launchctl kickstart → if restart fails → Discord alert to #status-and-health with diagnostic payload
  • Self-improvement: Logs response times per service. After 7 days, recalibrates “healthy” latency thresholds to actual P95 values. Eliminates false positives automatically.
  • Output: Writes health-log.md to Obsidian vault daily. Appends to rolling 30-day history.

Skill 02: Config Guardian

Resilience

Git-versioned config backup with automatic rollback on failure.

  • Trigger: Nightly (11:30 PM, before historian) or after any config change detected via fswatch
  • Scope: ~/.openclaw/, ~/.hermes/, ~/.n8n/, launchd plists, Caddy config, cron state
  • Action: git add -A && git commit with auto-generated diff summary. Push to local bare repo on NVMe.
  • Rollback: If System Health Probe detects failure within 10 min of a config commit → auto git revert HEAD → restart affected service → Discord alert with diff of what was rolled back.
  • Self-improvement: Tracks which config changes correlate with service failures. Builds a “dangerous keys” list. Warns before applying changes to those keys in the future.

Skill 03: Workflow Executor

Automation

Triggers n8n workflows on behalf of Jarvis with retry logic and result capture.

  • Trigger: Jarvis handoff message (see Handoff Protocol below) or scheduled cron
  • Execution: Calls n8n API (POST /api/v1/workflows/{id}/run) with payload. Monitors execution status via polling (GET /api/v1/executions/{id}). Timeout: 120s default, configurable per workflow.
  • Retry: Exponential backoff — 5s, 10s, 20s. Max 3 retries. If all fail, escalates to Jarvis with error payload.
  • Result capture: Writes execution result (success/fail, duration, output summary) to agent-shared/execution-log.md in Obsidian vault.
  • Self-improvement: Tracks workflow success rates over time. If a workflow drops below 80% success rate over 7 days, flags it for review and suggests common failure patterns.
Why These Three First

System Health Probe gives Hermes eyes. Config Guardian gives Hermes a safety net. Workflow Executor gives Hermes hands. Together they form the minimum viable executor: it can see what’s broken, protect against its own mistakes, and do useful work. Every subsequent skill builds on this foundation.

Handoff Protocol — How Jarvis Talks to Hermes

The handoff protocol is a JSON schema written as a markdown file to the shared Obsidian vault at /agent-shared/handoffs/. Jarvis writes the file. Hermes watches the directory via fswatch. When a new handoff appears, Hermes picks it up, executes it, and writes the result back. This pattern is borrowed from the OpenAI Agents SDK handoff model, adapted for file-based inter-process communication.

HANDOFF SCHEMA  ·  /agent-shared/handoffs/{timestamp}-{taskId}.md

---
taskId: "2026-04-10-pm-recap-trigger"
fromAgent: "jarvis"
toAgent: "hermes"
priority: "normal"           # critical | high | normal | low
skill: "workflow-executor"    # which Hermes skill to invoke
created: "2026-04-10T18:25:00-04:00"
deadline: "2026-04-10T18:35:00-04:00"  # hard timeout
status: "pending"             # pending → in_progress → completed | failed
---

## Task
Trigger the IXON PM Recap workflow in n8n.

## Context
- Workflow ID: abc123
- Expected output: Discord message to #ixon-evening-brief
- This is the first live test of the PM Recap since the DB patch fix.
- If it fails, capture the full error and report to Jarvis immediately.

## Parameters
```json
{
  "workflowId": "abc123",
  "timeout": 120,
  "retries": 3,
  "notifyOnComplete": true,
  "notifyChannel": "#status-and-health"
}
```

## Result (written by Hermes after execution)
status: completed
duration: 14.2s
output: "PM Recap delivered to #ixon-evening-brief at 6:31 PM ET"
retries_used: 0
errors: none

The handoff directory acts as a persistent, auditable task queue. Every handoff is a markdown file that Obsidian indexes, so you can search, graph-view, and trend your entire task execution history. Failed handoffs stay in the directory with status: failed for debugging. Completed ones get archived to /agent-shared/handoffs/archive/ weekly by Config Guardian.

Jarvis → Hermes

Command flow

  • Jarvis writes .md handoff file to /handoffs/
  • Hermes detects via fswatch (100ms latency)
  • Hermes reads file, validates schema, claims task (sets status: in_progress)
  • Hermes executes using the specified skill
  • Hermes writes result section back to file
  • Hermes posts completion/failure to Discord if notifyOnComplete: true

Hermes → Jarvis

Report flow

  • Hermes writes to /agent-shared/reports/ for non-task updates
  • Health check results, skill improvement suggestions, anomaly detections
  • Jarvis reads reports during nightly historian consolidation
  • Critical reports trigger immediate Discord notification from Hermes directly
  • Report format: same YAML frontmatter + markdown body

Mutual Self-Repair Loop

The self-repair loop is the immune system. Two agents watching each other, with launchd as the backup immune system underneath both. Three layers of recovery, each escalating only when the previous layer fails.

SELF-REPAIR CASCADE  ·  Three Layers

LAYER 1: SERVICE-LEVEL (launchd · 0-10 seconds)
├── KeepAlive: true on all plists
├── Process crashes → launchd restarts immediately
├── ThrottleInterval: 10s (prevents restart storms)
├── StandardOutPath + StandardErrorPath (all logs captured)
└── Handles: OOM kills, segfaults, unhandled exceptions

LAYER 2: CROSS-AGENT (Jarvis ↔ Hermes · 30-60 seconds)
├── Hermes monitors Jarvis:
│   Health check every 30s → port 18789 TCP + HTTP 200
│   2 consecutive failures → launchctl kickstart -k user/com.openclaw.gateway
│   Post recovery event to Discord #status-and-health
│
├── Jarvis monitors Hermes:
│   Cron check every 60s → verify Hermes PID alive + last heartbeat < 2min
│   2 consecutive failures → launchctl kickstart -k user/com.hermes.agent
│   Post recovery event to Discord #status-and-health
│
└── Heartbeat protocol:
    Each agent writes timestamp to /agent-shared/heartbeats/{agent}.json every 30s
    Format: {"agent":"hermes","ts":"2026-04-10T18:30:00Z","status":"healthy"}

LAYER 3: EMERGENCY (Claude/Cowork or Phil · minutes to hours)
├── Both agents down simultaneously (rare — requires system-level failure)
├── Discord webhook from healthchecks.io pings Phil
├── Phil opens Cowork session → Claude diagnoses via Bash tool
├── Or Phil SSHs in via Tailscale from phone
└── REBUILD.md on NVMe has step-by-step recovery procedure
Critical Rule

Never restart both agents simultaneously. If Hermes detects Jarvis is down, it restarts Jarvis and waits 30 seconds before checking its own health. If Jarvis detects Hermes is down, same pattern. This prevents a restart cascade where both agents keep killing and restarting each other. The 30-second cooldown is non-negotiable.

Reliability Engineering

Reliability isn’t hoping things work. It’s designing for the assumption that everything will fail, and building recovery into every layer. Here’s how to harden every service in the stack.

launchd Configuration — The Right Way

launchd is the single most important reliability layer. Every misconfigured plist is a ticking time bomb. These are the production-tested patterns.

01
KeepAlive: Conditional, Not Unconditional

Don’t use bare <true/>. Use dictionary form: SuccessfulExit: false restarts only on crashes (exit code ≠ 0), not clean shutdowns. This prevents phantom restarts during system updates or manual stops.

Exception: Hermes and OpenClaw watchdogs should use unconditional KeepAlive since they must always be running.

02
ThrottleInterval: Respect the 10-Second Rule

If your process exits within 10 seconds of launch, launchd enters throttle mode and delays the next restart. Design all services to stay alive for at least 10 seconds. If a health check script runs instantly, add a sleep 10 at the end. This is not optional — launchd will log “Service only ran for N seconds. Pushing respawn out.”

03
Never Daemonize Inside the Process

If your service calls daemon() or fork+exit, launchd thinks the parent died and tries to restart it. Run all services in the foreground. Let launchd be the process manager.

04
Log Everything

Every plist must have StandardOutPath and StandardErrorPath pointing to ~/.jarvis/logs/{service}-stdout.log and stderr.log. Without these, you’re debugging blind after any crash.

05
Startup Dependency Chains via Health Checks

launchd doesn’t have explicit dependency ordering. Instead, use wrapper scripts: OpenClaw’s startup script polls Ollama’s /api/tags endpoint before launching. Hermes polls OpenClaw’s port before starting its monitoring loop. This ensures correct startup order after cold boot without fragile sleep timers.

n8n — From Hobby to Production

!
Set N8N_ENCRYPTION_KEY Immediately

Generate with openssl rand -base64 32. Store in macOS Keychain. Set as environment variable in the launchd plist via EnvironmentVariables dict. Without this, all stored credentials (Google OAuth, Discord token, API keys) are stored with a default key that anyone with filesystem access can decrypt. This is the highest-priority n8n fix.

02
Migrate SQLite → PostgreSQL

SQLite locks the entire database on every write. Cannot be safely backed up while n8n is running. Incompatible with queue mode. Fine for experimentation, dangerous for production workflows that handle your IXON deal intelligence. PostgreSQL via Homebrew: brew install postgresql@16, configure n8n with DB_TYPE=postgresdb. Migration path is documented in n8n docs.

03
Error Workflows with Discord Alerts

Every workflow should have an error branch that catches failures, formats a diagnostic message, and posts to #status-and-health via Discord webhook. Use exponential backoff on retryable steps: 5s, 10s, 20s, max 3 retries. The AM/PM briefs were silently failing for days — this ensures that never happens again.

04
Bind to 127.0.0.1, Not 0.0.0.0

n8n should only accept connections from localhost. External access goes through Caddy reverse proxy with proper authentication. This eliminates the entire class of “someone on my network can hit my workflow engine” vulnerabilities.

Ollama — Keeping Inference Stable

01
Pin OLLAMA_KEEP_ALIVE to 24h

Default is 5 minutes — model unloads after inactivity, causing cold-start latency on next request. Set OLLAMA_KEEP_ALIVE=24h in the Ollama launchd plist environment. Your 36GB has headroom for persistent model loading.

02
Pin Model Version by Digest

You pulled mistral-small3.1:latest. If Ollama updates the tag, your inference behavior changes silently. Run ollama show mistral-small3.1 --format json and record the digest hash. Document it in REBUILD.md. If behavior changes, you can re-pull the exact version.

03
Health Check: Latency Probe, Not Just Connectivity

Don’t just check if Ollama is listening. Send a benchmark prompt (“Respond with OK”) and measure latency. If latency exceeds P95 threshold (calibrated after 7 days of data from Skill 01), trigger model reload. Ollama can enter degraded states where it accepts connections but inference is thrashing.

04
Memory Budget: Reserve 20GB for Models

Mistral Small 3.1 24B Q4_K_M needs ~15GB for weights plus KV cache overhead. Reserve 20GB for Ollama. That leaves ~16GB for OS + n8n + OpenClaw + Hermes + headroom. Monitor with memory_pressure command — if you hit “WARN” level, Ollama is overallocating.

Security Posture

The current security posture has critical gaps. OpenClaw binds to 0.0.0.0 with allowInsecureAuth: true. n8n has no encryption key. The 4TB NVMe is unencrypted. API keys live in config files. Here’s the hardening plan, prioritized by risk.

Zero-Trust Architecture

The principle: never trust any component, even on localhost. Every service authenticates to every other service. Every port is locked down. Every secret is encrypted.

!
Bind All Services to 127.0.0.1

OpenClaw: Change allowInsecureAuth: false, bind to loopback. This is the single biggest security vulnerability in the stack right now. Anyone on your home network can currently hit the gateway with no auth.

n8n: Bind to 127.0.0.1:5678. Already has basic auth, but should not be network-accessible.

Ollama: Already localhost-only. Verified.

!
Deploy Caddy as Reverse Proxy

Install via Homebrew: brew install caddy. Configure as the sole entry point for any service that needs external access. Caddy auto-generates TLS certs for local domains. Put n8n, OpenClaw, and any future web UIs behind Caddy with authentication middleware. This gives you HTTPS on all internal services with one config file.

03
Deploy Little Snitch for Outbound Control

The built-in macOS firewall only blocks inbound. Little Snitch ($30) gives you per-process outbound visibility. You’ll see exactly which services are phoning home, and can block anything unexpected. Particularly important for Ollama (should never make outbound connections) and n8n (should only hit googleapis.com, discord.com, api.anthropic.com, openrouter.ai).

Encryption — At Rest and In Transit

!
Encrypt the 4TB NVMe

Reformat as APFS (Encrypted). Apple Silicon hardware acceleration means zero performance impact. Store the password in macOS Keychain for auto-unlock. This protects your Ollama models, Obsidian vault, backups, and any sensitive data if the drive is physically stolen or the enclosure fails and gets RMA’d. Do this before storing any more data.

02
Verify FileVault on Internal SSD

Should be enabled by default on Apple Silicon. Verify with fdesetup status. If not enabled, enable it. Backup the recovery key to somewhere other than iCloud (1Password, printed in a safe, etc.).

03
TLS for All Internal Services

Caddy reverse proxy handles this automatically. Once deployed, all service-to-service communication goes through HTTPS. Self-signed CA from Caddy gets added to your system trust store. No more plaintext HTTP between services.

API Key Management

!
Move All Secrets to macOS Keychain

Every API key currently lives in a JSON config file. Move them to Keychain using security add-generic-password. Create a startup script that retrieves keys and exports as environment variables. Tools like envchain or ks simplify this. The keys in config files get replaced with placeholders or removed entirely.

02
Separate Keys Per Service

OpenClaw and Hermes should have different Anthropic API keys. n8n should have its own. If one key is compromised, you revoke only that key. Per-key usage tracking in Anthropic’s dashboard shows you exactly which service is spending what. Create keys named: anthropic-jarvis-prod, anthropic-hermes-prod, anthropic-n8n-prod.

03
90-Day Key Rotation

Set a quarterly calendar reminder (Jarvis can do this via n8n + Google Calendar). Rotation process: generate new key → update Keychain → restart affected service → verify service works → revoke old key. Zero-downtime rotation since most APIs support multiple active keys.

Disaster Recovery

The goal: if the Mac Studio catches fire tomorrow, you’re back online within 60 minutes on any new Mac with access to the 4TB NVMe. If the NVMe also dies, you’re back within 4 hours from cloud backups. Everything else is unacceptable.

3-2-1 Backup Strategy

Copy 1: Production

Internal SSD (configs, plists, workspace) + 4TB NVMe (models, vault, backups). This is the live system. Not a backup — it’s what you’re protecting.

Copy 2: Local Mirror

Nightly rsync from internal SSD configs to /Volumes/Jarvis/backups/ on NVMe. Hard-link incremental snapshots. 7-day rotation. Separate media (external vs internal). Handles SSD failure.

Copy 3: Cloud Offsite

Weekly encrypted push to Backblaze B2 ($6/TB/month) via rclone. Configs, vault, n8n database. Not models (re-downloadable). Handles fire, theft, or both devices dying simultaneously.

Nightly Backup Script

NIGHTLY BACKUP  ·  Cron 11:00 PM ET (before historian)

#!/bin/bash
# Backup target with date-based snapshot
BACKUP_DIR="/Volumes/Jarvis/backups/$(date +%Y-%m-%d)"
LATEST="/Volumes/Jarvis/backups/latest"

# Incremental rsync with hard links to previous backup
rsync -a --delete \
  --link-dest="$LATEST" \
  ~/.openclaw/ \
  ~/.hermes/ \
  ~/.n8n/ \
  ~/Library/LaunchAgents/com.openclaw.* \
  ~/Library/LaunchAgents/com.hermes.* \
  ~/Library/LaunchAgents/com.n8n.* \
  "$BACKUP_DIR/"

# Update latest symlink
ln -sfn "$BACKUP_DIR" "$LATEST"

# Prune backups older than 7 days
find /Volumes/Jarvis/backups/ -maxdepth 1 -type d -mtime +7 -exec rm -rf {} +

# Verify backup integrity
if [ -d "$BACKUP_DIR/openclaw" ]; then
  echo "$(date): Backup successful - $BACKUP_DIR" >> /Volumes/Jarvis/backups/backup.log
else
  # Alert via Discord webhook
  curl -s -X POST "$DISCORD_WEBHOOK" \
    -H "Content-Type: application/json" \
    -d '{"content":"⚠️ **BACKUP FAILED** - '"$(date)"' - Check backup.log"}'
fi

Interoperability — Everything Talks to Everything

The power of the system isn’t any single component — it’s how they connect. Every integration point needs a defined protocol, not ad-hoc wiring.

MCP Server — n8n Workflows as Tools

The n8n MCP server exposes your 5 workflows as callable tools for any MCP client (Claude Code, Cowork, future agents). Today you patched the SQLite DB to enable this. Going forward, the MCP server should be the primary way external agents interact with n8n.

01
Tag Workflows for Safety

Add tags to n8n workflows: agent-safe for workflows that can be triggered by any agent, manual-only for workflows that should only run via human trigger. MCP server should filter to only expose agent-safe tagged workflows.

02
Authenticate the MCP Server

Currently unauthenticated. Add bearer token auth. Each MCP client (Jarvis, Hermes, Claude) gets its own token. Log which client triggered which workflow.

03
Implement Timeout Budgets

MCP tool calls need explicit timeouts. AM/PM briefs can take 30-60 seconds (Gmail API + Claude synthesis). Set per-workflow timeouts in the MCP config. If a tool call exceeds its budget, return a structured timeout error that the caller can retry or escalate.
